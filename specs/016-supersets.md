# Specification: Суперсеты в логе тренировки

**Epic:** E2 — Учёт тренировок
**User Story:** Во время активной тренировки пользователь должен иметь возможность объединить два упражнения в суперсет — пару, выполняющуюся без отдыха между упражнениями. Это позволяет логировать продвинутые техники тренировки (антагонисты, push/pull-пары) и видеть структуру тренировки при просмотре истории.
**Related specs:** [005-workout-tracking.md](005-workout-tracking.md), [006-ai-workout-generator.md](006-ai-workout-generator.md)

---

## 1. User Goal

Суперсет — самая распространённая «продвинутая» техника силовой тренировки: два упражнения подряд без отдыха между ними. Обычно на антагонистах (бицепс + трицепс), либо push/pull-парах (жим + тяга), либо на верх/низ.

Сейчас в нашем приложении нет способа отметить такую связку: каждое упражнение и его подходы рендерятся независимым блоком. Это решение для:

1. **Логировать суперсет на месте** — отметить, что упражнение A и B выполняются как пара, не отдельно.
2. **Видеть структуру при просмотре** — открыл прошлую тренировку, видишь, что было «жим + тяга супер, 3 круга».
3. **Корректный таймер отдыха** — после A1 не нужен полный отдых, нужен короткий transition к B1.

Не-цели:
- Трисеты, гигантские сеты, циркуиты (REQ-04 закрывает: максимум 2 упражнения в группе; БД-поле позволяет больше, ограничивает UI).
- Суперсеты в плане тренировок (composer/recommender не знает про суперсеты; читай: всё, что добавляется сейчас — только в `exercise_logs`, не в `plan_exercises`).
- Отдельный «таймер суперсета» с автопереключением между упражнениями (будем смотреть, как зайдёт основной флоу).

---

## 2. Context

Самый компактный способ выразить «два лога принадлежат одной группе» — добавить nullable колонку `superset_group_id` (UUID) в `exercise_logs`. Все логи с одинаковым `superset_group_id` в одной тренировке считаются группой.

Идемпотентность для оффлайн (spec 015): `client_id` уже есть на каждом логе; `superset_group_id` тоже выдаётся клиентом (UUID v4 в момент группировки) — два устройства не сделают противоречивые группировки одного лога, потому что лог уникален по `client_id`.

UI принимает решение об «объединить» / «разъединить» — это локальная мутация существующих логов (PATCH соответствующих `exercise_logs`). Создание нового лога внутри уже-сформированного суперсета берёт `superset_group_id` от соседнего лога этого упражнения в этой же тренировке (если есть).

## 3. User Scenarios

### Scenario 1 — Объединить два упражнения в суперсет

Пользователь делает активную тренировку. Уже залогировал 1 подход «Жима лёжа» и 1 подход «Тяги штанги в наклоне».

**How to test independently**: Создать workout с двумя exercise_log'ами разных упражнений, вызвать `PATCH /exercise_logs/group` с парой id, проверить что у обоих появилось `superset_group_id`.

**Acceptance criteria**:

1. **Given** в активной тренировке два упражнения A и B, оба с ≥1 подходом, оба не в суперсете.
   **When** пользователь long-press на блоке упражнения A → меню «Объединить со следующим» → ткает B.
   **Then** все логи A и B получают одинаковый `superset_group_id`. UI показывает оба блока внутри одной рамки с маркером суперсета (полоса слева + значок 🔗 + надпись «Суперсет A+B»).

2. **Given** упражнение A добавлено в суперсет с B.
   **When** пользователь добавляет новый подход (A2) после A1 / B1.
   **Then** новый лог автоматически получает `superset_group_id` тот же. Подходы внутри пары следуют «змейкой»: A1 → B1 → A2 → B2 → A3 → B3 (set_number у A и B нумеруется независимо).

### Scenario 2 — Разъединить суперсет

Пользователь передумал, хочет вернуть упражнения в обычный режим.

**Acceptance criteria**:

1. **Given** в активной тренировке два упражнения объединены в суперсет (одинаковый `superset_group_id`).
   **When** long-press на любом блоке → «Разъединить».
   **Then** у всех логов A и B этой группы `superset_group_id` становится `null`. UI убирает рамку и значок.

### Scenario 3 — Просмотр истории с суперсетом

Прошлая тренировка содержала суперсет — пользователь хочет это увидеть.

**Acceptance criteria**:

1. **Given** в БД есть workout, где два упражнения имеют общий `superset_group_id`.
   **When** пользователь открывает экран деталей этой тренировки (`/training/view/:id`).
   **Then** упражнения отрендерены в одной карточке с маркером «Суперсет», показаны их подходы в порядке выполнения, после пары — общий комментарий «отдых X сек».

### Scenario 4 — Suggested pairs (опц., не в M1)

Когда пользователь логирует упражнение, и в каталоге есть «классическая пара» (например, жим лёжа ↔ тяга в наклоне), UI может предложить «Объединить с тягой?» Это поведение не входит в M1, будет рассмотрено отдельно.

## 4. Functional Requirements

- **REQ-01.** Колонка `exercise_logs.superset_group_id` (UUID nullable). Все логи с одинаковым `superset_group_id` в одной тренировке формируют группу.
- **REQ-02.** Сервер не валидирует количество упражнений в группе (БД позволяет 2+); UI ограничивает 2. Это даёт лёгкое расширение до трисетов в будущем, не требуя миграции.
- **REQ-03.** Эндпоинт `POST /workouts/{id}/supersets/group` принимает `{exercise_a, exercise_b}` (UUID упражнений из каталога) → атомарно проставляет `superset_group_id = uuid_generate_v4()` всем логам этих упражнений в этой тренировке.
- **REQ-04.** Эндпоинт `POST /workouts/{id}/supersets/ungroup` принимает `{group_id}` → проставляет `null` всем логам с этим `superset_group_id` в этой тренировке.
- **REQ-05.** При создании нового лога (`POST /workouts/{id}/logs`) сервер автоматически проставляет `superset_group_id` от существующего лога этого же упражнения в той же тренировке (если он есть и в суперсете). Это даёт UX «продолжать суперсет без явной группировки каждого подхода».
- **REQ-06.** При удалении лога — не влияет на `superset_group_id` других логов группы.
- **REQ-07.** API возвращает `superset_group_id` в `ExerciseLogRead`. UI клиента группирует логи по этому полю при рендере.

---

## 5. Non-functional Requirements

- **NFR-01.** Группировка/разгруппировка отвечает <100 мс на 95-перцентиле (одна UPDATE-операция).
- **NFR-02.** Старые workout-записи (до миграции 0022) имеют `superset_group_id = NULL` — рендерятся как раньше, без рамок.

## 6. Data Model

```sql
-- migration 0022_supersets.py
ALTER TABLE exercise_logs ADD COLUMN superset_group_id UUID NULL;
CREATE INDEX ix_exercise_logs_superset_group
  ON exercise_logs(superset_group_id)
  WHERE superset_group_id IS NOT NULL;
```

Семантика: в рамках одного `workout_id` все строки с одинаковым `superset_group_id` — одна группа. В разных workout'ах — даже если совпадение `superset_group_id` (UUID-коллизия не предусмотрена) — это были бы разные группы, потому что объединение всегда в контексте одной тренировки.

## 7. Screens / UI

### Активная тренировка (`active_workout_screen`)
- Long-press на блоке упражнения → меню действий:
  - «Объединить со следующим» (показывается, если есть следующее upr и оба не в группе).
  - «Объединить с предыдущим» (аналогично).
  - «Разъединить» (показывается, если упражнение в группе).
- Объединённые упражнения окружены тонкой рамкой с blue/orange левой полосой (Portal-стиль), сверху бейдж «🔗 Суперсет».
- Новые подходы добавляются в любое упражнение группы — система автоматически проставляет `superset_group_id`.

### Детали тренировки (`workout_detail_screen`)
- Группы рендерятся одним блоком, как на active screen.
- Подходы показаны в порядке `logged_at`.

## 8. API Contracts

### Группировка

```
POST /api/v1/workouts/{workout_id}/supersets/group
{
  "exercise_ids": ["<uuid_a>", "<uuid_b>"]
}
→ 200 OK
{
  "group_id": "<new-uuid>",
  "logs_updated": 6
}
```

### Разгруппировка

```
POST /api/v1/workouts/{workout_id}/supersets/ungroup
{
  "group_id": "<uuid>"
}
→ 200 OK
{
  "logs_updated": 6
}
```

### ExerciseLogRead (расширение)

```jsonc
{
  "id": "<uuid>",
  ...
  "superset_group_id": "<uuid>" | null
}
```

## 9. Edge Cases

- **Один лог в группе.** Сервер не запрещает (например, после удаления одного из упражнений-партнёров). UI рендерит такой случай как обычный одиночный блок без бейджа, но `superset_group_id` сохраняется на случай добавления подхода — он будет привязан к старой группе. Это безопасно: при обнаружении единственного `set_member` UI просто скрывает рамку.
- **Группировка трёх упражнений вручную через API.** REQ-02 разрешает; UI просто не предлагает действие. Будущие версии могут включить трисеты без изменения схемы.
- **Объединение упражнений из разных workout-ов.** Невозможно: эндпоинт принимает `workout_id` в URL и работает только с логами этой тренировки.
- **Конфликт offline (два устройства группируют по-разному).** Атомарность через PATCH каждой строки `exercise_logs`. Идемпотентность — через `client_id` на самих логах. Если два устройства независимо нажали «Объединить» — победит последний sync, перезатёрта `superset_group_id`.

## 10. Success Criteria

- **SC-01.** Тест: создать workout с двумя упражнениями, вызвать `/supersets/group`, проверить что все логи получили одинаковый id и индекс на нём работает.
- **SC-02.** Тест: вызвать `/supersets/ungroup`, проверить что id сброшен.
- **SC-03.** На активной тренировке long-press → «Объединить» → видимая рамка с маркером 🔗. После добавления нового подхода в группу — он автоматически получает `superset_group_id`.
- **SC-04.** На экране деталей тренировки группа отрисована как один блок.
- **SC-05.** Старые workout'ы (до миграции 0022) рендерятся как раньше, без падений.

## 11. Open Questions

- **Q1.** Нужен ли свой таймер отдыха внутри пары (например, 15-20с transition вместо 90с)? Сейчас нет — все подходы используют общий `rest_seconds`. Перенесём в следующую версию.
- **Q2.** В разделе «Аналитика» нужно ли отдельно считать тоннаж суперсетов? Нет — суперсет это техника выполнения, а не отдельный объём; общий тоннаж тренировки тот же.
- **Q3.** Стоит ли композеру плана (spec 006) учитывать суперсеты? Не сейчас (не-цель). При расширении — composer мог бы помечать `plan_exercises.superset_group_id`, а active workout наследовал бы их.
