# Specification: Регистрация и авторизация

**Epic:** E1 — Базовая работа пользователя
**User Story:** Пользователь должен иметь персональный аккаунт, чтобы его данные о теле, тренировках и плане сохранялись и были доступны только ему.
**Related specs:** [002-user-profile.md](002-user-profile.md)

---

## 1. User Goal

Пользователь приходит в приложение впервые и хочет создать аккаунт, чтобы начать вести учёт тренировок и InBody-показателей. Он также должен иметь возможность вернуться в приложение после удаления/переустановки и восстановить доступ — даже если забыл пароль. Без аккаунта приложение не работает.

---

## 2. Context

InBody-показатели и история тренировок — персональные данные. По требованию GDPR (NFR-проекта) они должны быть привязаны к идентифицируемому пользователю и зашифрованы. Email используется как первичный идентификатор и канал служебной коммуникации (подтверждение, восстановление пароля).

---

## 3. User Scenarios

### Scenario 1 — Регистрация по email

Новый пользователь создаёт аккаунт, указав email и пароль. После регистрации на email приходит письмо со ссылкой для подтверждения.

**How to test independently**: Открыть приложение в incognito, пройти форму регистрации с валидным email, проверить что письмо доставлено и ссылка активирует аккаунт.

**Acceptance criteria**:

1. **Given** пользователь на экране регистрации.
   **When** он вводит валидный email, пароль (≥8 символов, минимум 1 цифра и 1 буква), подтверждение пароля и нажимает «Создать аккаунт».
   **Then** аккаунт создаётся со статусом `email_unconfirmed`, отправляется письмо с verification token, пользователь видит экран «Проверьте почту».

2. **Given** пользователь ввёл email, который уже занят.
   **When** он нажимает «Создать аккаунт».
   **Then** показывается ошибка «Аккаунт с этим email уже существует. Войти?» с CTA на экран логина.

3. **Given** пользователь ввёл слабый пароль.
   **When** он пытается отправить форму.
   **Then** форма не отправляется, рядом с полем пароля подсвечивается, какому требованию не соответствует.

### Scenario 2 — Подтверждение email

Пользователь переходит по ссылке из письма, его аккаунт активируется.

**How to test independently**: Зарегистрироваться, перейти по ссылке из письма, проверить статус аккаунта.

**Acceptance criteria**:

1. **Given** аккаунт со статусом `email_unconfirmed` и валидный verification token.
   **When** пользователь открывает ссылку из письма.
   **Then** статус аккаунта меняется на `active`, пользователь автоматически логинится и попадает на экран онбординга профиля.

2. **Given** verification token истёк (>24 часа) или уже использован.
   **When** пользователь открывает ссылку.
   **Then** показывается экран с ошибкой и кнопка «Отправить новое письмо».

3. **Given** аккаунт ещё не подтверждён.
   **When** пользователь пробует залогиниться.
   **Then** показывается экран «Подтвердите email» с кнопкой «Отправить письмо повторно».

### Scenario 3 — Авторизация

Зарегистрированный пользователь логинится в приложение.

**How to test independently**: Создать аккаунт, выйти, войти снова через email + пароль.

**Acceptance criteria**:

1. **Given** активный аккаунт.
   **When** пользователь вводит правильные email и пароль.
   **Then** выдаётся session token, пользователь попадает на главный экран.

2. **Given** активный аккаунт.
   **When** пользователь ввёл неправильный пароль.
   **Then** показывается ошибка «Неверный email или пароль» (без указания, что именно не так — для защиты от перебора).

3. **Given** 5 неуспешных попыток входа за 10 минут с одного IP/email.
   **When** пользователь пробует войти ещё раз.
   **Then** вход блокируется на 15 минут, показывается ошибка «Слишком много попыток. Попробуйте через 15 минут».

### Scenario 4 — Восстановление пароля

Пользователь забыл пароль и хочет его сбросить.

**How to test independently**: На экране логина нажать «Забыли пароль?», получить письмо, перейти по ссылке, задать новый пароль, войти с ним.

**Acceptance criteria**:

1. **Given** пользователь на экране логина.
   **When** он нажимает «Забыли пароль?» и вводит email.
   **Then** на email отправляется письмо со ссылкой сброса (валидной 1 час), пользователь видит «Письмо отправлено» (даже если email не зарегистрирован — для защиты от утечки).

2. **Given** пользователь перешёл по ссылке сброса.
   **When** он вводит новый пароль и подтверждение.
   **Then** пароль обновляется, все активные session-токены отзываются, пользователь логинится с новым паролем.

3. **Given** ссылка сброса истекла.
   **When** пользователь её открывает.
   **Then** показывается «Ссылка истекла» с кнопкой запросить новую.

### Scenario 5 — Выход из аккаунта

Пользователь выходит из аккаунта вручную.

**How to test independently**: Войти, нажать «Выйти», убедиться что доступ к приложению закрыт.

**Acceptance criteria**:

1. **Given** залогиненный пользователь.
   **When** он нажимает «Выйти» в настройках.
   **Then** session token отзывается, пользователь возвращается на экран логина.

## 4. Functional Requirements

| #      | Requirement                                                                                          |
|--------|------------------------------------------------------------------------------------------------------|
| REQ-01 | Регистрация по email + пароль                                                                        |
| REQ-02 | Авторизация по email + пароль                                                                        |
| REQ-03 | Подтверждение email через ссылку с токеном (TTL 24 часа)                                             |
| REQ-04 | Восстановление пароля через email-ссылку с токеном (TTL 1 час)                                       |
| REQ-05 | Хранение пароля в виде хэша (bcrypt/argon2), не в открытом виде                                      |
| REQ-06 | Rate limiting на попытки входа: 5 неудач за 10 минут → блок на 15 минут                              |
| REQ-07 | Session token с TTL и refresh-механизмом                                                             |
| REQ-08 | Logout с отзывом текущего session token                                                              |
| REQ-09 | При смене пароля все активные сессии отзываются                                                      |
| REQ-10 | Валидация пароля: ≥8 символов, минимум 1 цифра и 1 буква                                             |
| REQ-11 | Удаление аккаунта (GDPR): пользователь может запросить удаление, все его данные удаляются за ≤30 дней |

---

## 5. Non-functional Requirements

| #      | Requirement                                                              | Category    |
|--------|--------------------------------------------------------------------------|-------------|
| NFR-01 | Email пароля и токена шифруются в БД (at-rest encryption)                | Security    |
| NFR-02 | Все auth-эндпоинты только по HTTPS                                       | Security    |
| NFR-03 | Ответ на `/auth/login` ≤500 ms на 95-перцентиле                          | Performance |
| NFR-04 | На запросы по несуществующему email отвечать одинаково (timing-safe)     | Security    |

## 6. Data Model

**Entity: User**

| Field              | Type      | Constraints                                | Description                               |
|--------------------|-----------|--------------------------------------------|-------------------------------------------|
| id                 | UUID      | PK                                         | Уникальный идентификатор                  |
| email              | String    | Unique, required, format=email             | Логин + канал писем                       |
| password_hash      | String    | Required                                   | bcrypt/argon2 хэш                         |
| email_status       | Enum      | `unconfirmed` \| `active`, default=unconfirmed | Подтверждён ли email                  |
| created_at         | Timestamp | Required, immutable                        | Время регистрации                         |
| deleted_at         | Timestamp | Optional                                   | Soft-delete для GDPR-flow                 |

**Entity: AuthToken** (verification, password reset, session)

| Field        | Type      | Constraints                                                             | Description                          |
|--------------|-----------|-------------------------------------------------------------------------|--------------------------------------|
| id           | UUID      | PK                                                                      |                                      |
| user_id      | UUID      | FK → User, required                                                     |                                      |
| type         | Enum      | `email_verify` \| `password_reset` \| `session` \| `refresh`            |                                      |
| token_hash   | String    | Unique, required                                                        | Хэш токена (raw токен в письме)      |
| expires_at   | Timestamp | Required                                                                |                                      |
| used_at      | Timestamp | Optional                                                                | Для одноразовых токенов              |

## 7. Screens

### Screen: Регистрация

**Purpose**: Создание аккаунта.

**Elements**:
- Поле email
- Поле пароль (с toggle «показать»)
- Поле подтверждения пароля
- Чекбокс «Согласен с условиями и политикой конфиденциальности» (обязателен для GDPR)
- Кнопка «Создать аккаунт»
- Ссылка «Уже есть аккаунт? Войти»

**States**:
- Loading: спиннер на кнопке после нажатия
- Error: текст ошибки под полем
- Success: переход на экран «Проверьте почту»

**Navigation**: Login → Регистрация → Проверьте почту → (по ссылке из письма) → Онбординг профиля.

### Screen: Логин

**Elements**:
- Поле email, поле пароль
- Кнопка «Войти»
- Ссылки «Забыли пароль?», «Создать аккаунт»

**States**:
- Loading, Error, Success → главный экран
- Locked (rate limit): «Попробуйте через 15 минут»

### Screen: Восстановление пароля (запрос)

**Elements**: Поле email, кнопка «Отправить ссылку».

### Screen: Восстановление пароля (ввод нового)

**Elements**: Поле новый пароль, подтверждение, кнопка «Сохранить».

## 8. API Endpoints

### Регистрация

```
POST /api/v1/auth/register
```

**Request**:
```json
{ "email": "user@example.com", "password": "string" }
```

**Response 201**:
```json
{ "user_id": "uuid", "email_status": "unconfirmed" }
```

**Errors**:

| Code | Situation                | Response                                                       |
|------|--------------------------|----------------------------------------------------------------|
| 400  | Слабый пароль            | `{"error": "weak_password", "message": "..."}`                 |
| 409  | Email уже занят          | `{"error": "email_taken", "message": "..."}`                   |

### Подтверждение email

```
POST /api/v1/auth/verify-email
{ "token": "string" }
```

### Логин

```
POST /api/v1/auth/login
{ "email": "...", "password": "..." }
```

**Response 200**:
```json
{ "access_token": "...", "refresh_token": "...", "expires_in": 3600 }
```

**Errors**: 401 `invalid_credentials`, 403 `email_unconfirmed`, 429 `rate_limited`.

### Восстановление пароля

```
POST /api/v1/auth/forgot-password   { "email": "..." }
POST /api/v1/auth/reset-password    { "token": "...", "new_password": "..." }
```

### Logout

```
POST /api/v1/auth/logout    (с access token в заголовке)
```

### Удаление аккаунта (GDPR)

```
DELETE /api/v1/auth/account    (с подтверждением паролем)
```

## 10. Edge Cases

- Пользователь заблокирован (rate limit) → корректный ответ 429 с `retry_after` в секундах.
- Подтверждение email по уже использованной ссылке → 410 `token_used`.
- Refresh token украден → отзыв всех сессий по запросу пользователя из настроек.
- Пользователь меняет email — **out of scope** в MVP, выносится в отдельную story.

---

## 11. Out of Scope

- OAuth (Google/Apple/VK) — отдельная story
- Двухфакторная аутентификация (2FA)
- Смена email
- B2B-кабинет тренера / организации (см. overview, открытый вопрос #3)
- SSO для фитнес-клубов
