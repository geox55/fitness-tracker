# Specification: Адаптация плана и чат-ассистент

**Epic:** E4 — AI-блок Маши
**User Story:** Пользователь должен получать автоматические корректировки плана при изменении веса/цели/частоты, а также мочь задать вопрос ИИ-ассистенту в чате (мотивация, рекомендации).
**Related specs:** [002-user-profile.md](002-user-profile.md), [003-inbody-measurements.md](003-inbody-measurements.md), [006-ai-workout-generator.md](006-ai-workout-generator.md), [007-ai-nutrition-planner.md](007-ai-nutrition-planner.md), [008-ai-inbody-predictor.md](008-ai-inbody-predictor.md), [011-notifications.md](011-notifications.md)

---

## 1. User Goal

Пользователь не должен сам решать, когда пора пересобрать план. Если он изменил цель, увеличил частоту, потерял или набрал вес заметно — приложение замечает это и либо автоматически пересобирает планы, либо предлагает пересобрать. Также пользователь иногда хочет «обсудить» что-то — задать вопрос «почему мне дали такие подходы?», получить мотивационное сообщение.

Эта спека покрывает два связанных подмодуля Маши:
1. **Plan adaptation** — детерминированные правила и rule-based логика, которые могут вызывать ML-блоки ([006](006-ai-workout-generator.md) или [007](007-ai-nutrition-planner.md)) для регенерации.
2. **Chat-assistant** — диалог с пользователем. **Не своя NLP-модель**: scripted-flow с FAQ + опциональный LLM-fallback (см. open question #1 в overview).

---

## 2. Context

Адаптация — это сервис-watcher, который слушает события:
- Новый InBody-замер → проверяет дельту с предыдущим.
- Изменение поля профиля (`goal`, `training_frequency`, `equipment_available`).
- Прошло ≥4 недели с генерации плана.

При срабатывании триггера — либо автоматически перегенерирует, либо ставит флаг `plan_rebuild_required` и шлёт уведомление.

Чат — отдельный модуль с тремя источниками ответов: scripted (FAQ), templated (на основе текущих данных пользователя), LLM (для свободных вопросов).

## 3. User Scenarios

### Scenario 1 — Автокоррекция при изменении веса

Пользователь добавил InBody-замер: вес изменился значительно (>2 кг или >3% от предыдущего за <30 дней).

**How to test independently**: Создать пользователя с активным планом, добавить новый InBody с дельтой 3 кг от последнего, проверить что план питания пересчитан и пользователь получил уведомление.

**Acceptance criteria**:

1. **Given** есть активный план и приходит новый InBody.
   **When** дельта `|new.weight_kg − prev.weight_kg| > 2 кг` ИЛИ изменение >3% за <30 дней.
   **Then** план питания ([007](007-ai-nutrition-planner.md)) перегенерируется автоматически; пользователь получает push/email «Ваш план питания обновлён» (см. [011](011-notifications.md)); план тренировок не трогается, но ставится флаг `plan_rebuild_recommended`.

2. **Given** дельта незначительна (<2 кг и <3%).
   **When** приходит новый InBody.
   **Then** план не меняется, обновляется только прогноз ([008](008-ai-inbody-predictor.md)).

### Scenario 2 — Регенерация при смене цели или частоты

Пользователь сменил цель с «похудение» на «набор массы».

**Acceptance criteria**:

1. **Given** активный план.
   **When** пользователь меняет в профиле `goal` или `training_frequency`.
   **Then** оба плана (тренировок и питания) помечаются `plan_rebuild_required: true`; пользователь видит баннер на главном экране «План устарел, перегенерировать?» с CTA.

2. **Given** пользователь подтверждает регенерацию.
   **When** он нажимает CTA.
   **Then** старые планы архивируются, новые генерируются по обновлённым параметрам.

3. **Given** пользователь игнорирует баннер 7 дней.
   **When** проходит фоновая проверка.
   **Then** регенерация запускается автоматически, пользователь получает уведомление.

### Scenario 3 — Окончание 4-недельного цикла

**Acceptance criteria**:

1. **Given** план тренировок: `valid_until` ≤ today.
   **When** проходит фоновая проверка (раз в сутки).
   **Then** план архивируется; ML-модель ([006](006-ai-workout-generator.md)) генерирует новый с учётом данных прошедшего цикла (фактических подходов/повторений из тренировок).

2. **Given** в прошлом цикле пользователь стабильно превышал целевые повторения.
   **When** генерируется новый план.
   **Then** в новом плане увеличены целевые веса/повторения (прогрессия от факта).

### Scenario 4 — Ассистент: мотивационное сообщение

Раз в неделю пользователь получает мотивационное сообщение в чате/уведомлении.

**Acceptance criteria**:

1. **Given** прошёл понедельник, у пользователя есть прогресс за прошлую неделю.
   **When** срабатывает scheduled job.
   **Then** в чате появляется новое сообщение от ассистента: краткое summary («ты сделал 3 тренировки, тоннаж +5%, продолжай») + одна персонализированная рекомендация.

2. **Given** в прошлой неделе тренировок было 0.
   **When** срабатывает scheduled job.
   **Then** генерируется поддерживающее сообщение («неделя без зала — бывает; вот 1 короткая домашняя тренировка для возвращения»).

### Scenario 5 — Ассистент: ответ на вопрос (scripted-flow)

Пользователь нажимает кнопку «Почему мне дали такие подходы?» из контекстного меню упражнения.

**Acceptance criteria**:

1. **Given** пользователь в чате.
   **When** он выбирает кнопку из quick replies («Почему такие подходы?», «Что такое RPE?», «Зачем мне белок?», ...).
   **Then** ассистент отвечает по шаблону, подставляя значения из текущего профиля и плана пользователя.

### Scenario 6 — Ассистент: свободный вопрос (LLM)

Пользователь печатает свободный текст: «можно ли мне есть мороженое?»

**Acceptance criteria**:

1. **Given** пользователь печатает в чат свободный текст.
   **When** скриптовый матчер не находит подходящий FAQ-ответ И LLM включён.
   **Then** запрос отправляется в LLM с контекстом (профиль, текущая цель, аллергии); ответ возвращается ≤15 секунд; помечается «персональный совет, не медицинская рекомендация».

2. **Given** LLM выключен или недоступен.
   **When** свободный вопрос не подходит к шаблонам.
   **Then** возвращается дефолтный ответ «Я пока не умею отвечать на это, попробуйте перефразировать или выберите тему ниже» + quick replies.

3. **Given** пользователь спрашивает что-то на чувствительную тему (медицинское, психическое здоровье).
   **When** ассистент детектирует ключевые слова.
   **Then** возвращается шаблон «Это вопрос к врачу/специалисту, я не могу его консультировать» + ссылка на дисклеймер.

## 4. Functional Requirements

### Адаптация плана

| #      | Requirement                                                                                                                |
|--------|----------------------------------------------------------------------------------------------------------------------------|
| REQ-01 | Watcher на изменение InBody.weight_kg: триггер при дельте >2 кг или >3% за <30 дней                                         |
| REQ-02 | Watcher на изменение полей профиля: `goal`, `training_frequency`, `equipment_available`, `allergies` → ставит `plan_rebuild_required` |
| REQ-03 | Watcher на окончание 4-недельного цикла: автоматически архивирует и регенерирует                                          |
| REQ-04 | Принудительная регенерация по флагу через 7 дней игнорирования                                                              |
| REQ-05 | Регенерация плана питания учитывает новый вес из последнего InBody                                                          |
| REQ-06 | Регенерация плана тренировок учитывает фактические данные предыдущего цикла (тоннаж, RPE)                                   |
| REQ-07 | Архивирование старого плана сохраняется в истории                                                                            |
| REQ-08 | Уведомление пользователю при автоматической регенерации (см. [011](011-notifications.md))                                  |

### Чат-ассистент

| #      | Requirement                                                                                                                 |
|--------|-----------------------------------------------------------------------------------------------------------------------------|
| REQ-09 | Чат с историей сообщений (UI thread)                                                                                         |
| REQ-10 | Quick replies — список заранее определённых тем/вопросов на главном экране чата                                              |
| REQ-11 | Scripted ответ на FAQ с подстановкой значений из профиля/плана                                                                |
| REQ-12 | Опциональный LLM-fallback для свободных вопросов (вкл/выкл через feature flag)                                                |
| REQ-13 | Детектор чувствительных тем (медицина, психика) → шаблонный ответ-disclaimer                                                  |
| REQ-14 | Еженедельные мотивационные сообщения (scheduled job, 1 раз/неделя)                                                            |
| REQ-15 | Ассистент не даёт медицинских рекомендаций (always disclaimer внизу LLM-ответа)                                              |

## 5. Non-functional Requirements

| #      | Requirement                                                                                          | Category    |
|--------|------------------------------------------------------------------------------------------------------|-------------|
| NFR-01 | Watcher срабатывает в рамках транзакции записи нового InBody/изменения профиля (без задержки)         | Reliability |
| NFR-02 | Scripted ответ ≤200 ms                                                                                 | Performance |
| NFR-03 | LLM ответ ≤15 секунд + индикатор «печатает»                                                            | UX          |
| NFR-04 | История чата хранится за пользователем; PII не логируется во внешних сервисах (LLM провайдер)         | Privacy     |
| NFR-05 | LLM-промпт фильтрует личные данные перед отправкой (анонимизация)                                     | Privacy     |

## 7. Screens

### Screen: Чат с ассистентом

**Purpose**: Диалог с ИИ + мотивационные сообщения.

**Elements**:
- Лента сообщений (типичный chat UI: bubble + время)
- Quick replies внизу (chips): «Почему такие подходы?», «Что съесть до тренировки?», «Я пропустил тренировку, что делать?», ...
- Поле свободного ввода (если LLM включён) или disabled (если выключен) с подписью «Выберите тему ниже»
- Disclaimer внизу: «Не медицинская рекомендация»

**States**:
- Loading: «печатает...»
- LLM offline: бейдж «Свободные вопросы временно недоступны»

### UI: Баннер «План устарел»

Показывается на главном экране, если `plan_rebuild_required = true`.

**Elements**:
- Текст: «Вы изменили цель — план тренировок и питания требует обновления»
- CTA: «Обновить» / «Позже»

## 9. API Endpoints

### Чат — список сообщений

```
GET /api/v1/chat/messages?limit=50&before=2026-04-28T00:00:00Z
```

### Отправить сообщение

```
POST /api/v1/chat/messages
{ "content": "что съесть до тренировки?" }
```

**Response 200**: пара сообщений (user + assistant) с `source` ответа.

**Errors**: 503 если LLM недоступен и скрипт не нашёл ответа (UI показывает дефолт).

### Quick replies

```
GET /api/v1/chat/quick-replies
```
**Response 200**: список тематических кнопок, персонализированных под текущий контекст пользователя.

### Триггеры адаптации (внутренние)

Эти эндпоинты вызываются другими сервисами, не пользователем:
```
POST /internal/v1/adaptation/check-weight    { "user_id": "..." }
POST /internal/v1/adaptation/check-cycles     (cron, без user_id, проходит по всем)
```

### Подтверждение регенерации (пользователем)

```
POST /api/v1/plans/rebuild
{ "target": "workout" | "nutrition" | "both" }
```

### События адаптации

```
GET /api/v1/adaptation/events?status=pending
```

## 10. Edge Cases

- Пользователь добавил два InBody подряд за один день → сравнивается с предыдущим из другого дня, не друг с другом.
- Изменение `goal` происходит несколько раз за день → debounce: rebuild_required ставится один раз, регенерация не запускается каждый раз.
- LLM возвращает ответ с явно неверными медицинскими утверждениями → системный промпт всегда добавляет disclaimer; модерация через системный промпт + перечень запрещённых тем.
- Пользователь спамит чат → rate limit: 30 сообщений/час.
- Пользователь удаляет аккаунт → история чата и события адаптации удаляются вместе с пользователем (GDPR).

## 11. Out of Scope

- Своя NLP-модель для чата (зафиксировано: scripted + LLM)
- Голосовой ввод
- Многоязычность чата (MVP — только русский)
- Чат между пользователями / с тренером — out of scope в MVP
- Push-уведомления внутри чата при ответе ассистента (только in-app в MVP)
- Активные «подталкивания» пользователя (например, через 3 дня без тренировки → push) — выносится в [011](011-notifications.md) фаза 2
