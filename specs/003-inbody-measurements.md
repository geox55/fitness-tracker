# Specification: InBody-измерения

**Epic:** E1 — Базовая работа пользователя
**User Story:** Пользователь должен иметь возможность фиксировать результаты InBody-анализа (загрузкой PDF или вручную) и видеть историю, чтобы отслеживать состав тела во времени и кормить им AI-блоки.
**Related specs:** [002-user-profile.md](002-user-profile.md), [008-ai-inbody-predictor.md](008-ai-inbody-predictor.md), [010-analytics-reports.md](010-analytics-reports.md), [013-inbody-pdf-import.md](013-inbody-pdf-import.md), [011-notifications.md](011-notifications.md)

---

## 1. User Goal

Пользователь периодически проходит InBody-анализ в зале или клинике. Он хочет загружать новые результаты в приложение (одной кнопкой через PDF или вручную, если PDF нет), видеть полную историю всех своих измерений и сравнивать их между собой. Эти данные должны быть основой для AI-генератора тренировок ([006](006-ai-workout-generator.md)) и AI-предиктора изменений ([008](008-ai-inbody-predictor.md)).

---

## 2. Context

InBody-протоколы выдают PDF с фиксированной структурой: вес, мышечная масса, жировая масса, % жира, висцеральный жир, вода в организме, BMR, и т.д. Парсинг PDF выделен в отдельную спеку ([013](013-inbody-pdf-import.md)) — здесь описывается только пользовательский flow и хранение записей.

Любая запись об измерении — append-only: пользователь не может редактировать прошлые измерения (только удалять явно), потому что они служат training data для ML и должны отражать историческую правду.

---

## 3. User Scenarios

### Scenario 1 — Ручной ввод InBody-измерения

Пользователь делал InBody, но PDF нет на руках — он вводит ключевые цифры вручную.

**How to test independently**: Зайти в раздел InBody → «Добавить измерение» → «Ввести вручную», заполнить поля, сохранить, проверить что запись появилась в истории.

**Acceptance criteria**:

1. **Given** пользователь на экране InBody.
   **When** он выбирает «Добавить вручную», вводит дату измерения, вес, % жира, мышечную массу (минимум) и сохраняет.
   **Then** запись сохраняется, отображается в списке истории, обновляются графики ([010](010-analytics-reports.md)).

2. **Given** пользователь не заполнил обязательные поля (дата + вес + % жира).
   **When** он нажимает «Сохранить».
   **Then** показывается ошибка с подсветкой пустых полей.

3. **Given** пользователь указал дату в будущем.
   **When** он пробует сохранить.
   **Then** показывается ошибка «Дата измерения не может быть в будущем».

4. **Given** пользователь указал дату, на которую уже есть измерение.
   **When** он сохраняет.
   **Then** показывается предупреждение «На эту дату уже есть запись. Заменить?» — выбор: заменить / отменить / сохранить как отдельную запись (с разным временем).

### Scenario 2 — Загрузка PDF InBody

Пользователь хочет быстро загрузить отчёт InBody без ручного ввода.

**How to test independently**: На экране InBody выбрать «Загрузить PDF», прикрепить файл с тестового стенда InBody, проверить, что данные распарсились и появилась запись.

**Acceptance criteria**:

1. **Given** пользователь на экране InBody.
   **When** он выбирает «Загрузить PDF» и прикрепляет валидный InBody PDF (≤10 MB).
   **Then** PDF отправляется на парсинг ([013](013-inbody-pdf-import.md)), пользователь видит экран превью с распознанными полями.

2. **Given** показан экран превью.
   **When** пользователь подтверждает.
   **Then** запись сохраняется как обычное измерение; в записи сохраняется `source = pdf` и ссылка на оригинал.

3. **Given** парсинг не смог извлечь обязательные поля.
   **When** пользователь видит экран превью с пропусками.
   **Then** он может дозаполнить вручную или отменить загрузку.

4. **Given** пользователь загрузил файл, который не InBody PDF.
   **When** парсер не находит характерных маркеров.
   **Then** показывается ошибка «Это не похоже на InBody-отчёт. Введите вручную?» с CTA на ручной ввод.

### Scenario 3 — История InBody

Пользователь хочет посмотреть, как менялись его показатели.

**How to test independently**: Создать 3+ измерения с разными датами, открыть «История InBody», проверить хронологический список.

**Acceptance criteria**:

1. **Given** у пользователя есть N измерений.
   **When** он открывает «История InBody».
   **Then** видит список, отсортированный по `measured_at` DESC, с ключевыми полями (дата, вес, % жира, мышечная масса).

2. **Given** пользователь тапает на запись.
   **When** карточка раскрывается.
   **Then** видит все поля измерения, источник (`pdf`/`manual`), кнопку «Скачать оригинал» (если PDF) и «Удалить».

3. **Given** у пользователя 0 измерений.
   **When** он открывает «История InBody».
   **Then** видит empty state с CTA «Добавить первое измерение».

### Scenario 4 — Удаление измерения

**Acceptance criteria**:

1. **Given** пользователь открыл карточку измерения.
   **When** он нажимает «Удалить» и подтверждает.
   **Then** запись удаляется (hard delete после подтверждения), графики и сравнения обновляются.

2. **Given** удалённая запись была единственной → AI-блоки откатываются на baseline-данные из профиля.

### Scenario 5 — Уведомление о повторном InBody

См. подробно в [011-notifications.md](011-notifications.md).

**Acceptance criteria**:

1. **Given** прошло >30 дней с последнего InBody-измерения.
   **When** проходит фоновая проверка (раз в сутки).
   **Then** пользователю отправляется email и in-app push «Пора повторить InBody-анализ».

## 4. Functional Requirements

| #      | Requirement                                                                                                |
|--------|------------------------------------------------------------------------------------------------------------|
| REQ-01 | Ручной ввод измерения с обязательными полями: дата, вес, % жира                                            |
| REQ-02 | Дополнительные поля при ручном вводе: мышечная масса, висцеральный жир, % воды, протеин, минералы, BMR, FFM, BMI |
| REQ-03 | Загрузка PDF (≤10 MB), парсинг и предзаполнение полей с возможностью корректировки                         |
| REQ-04 | Хранение источника (`pdf` \| `manual`) и оригинального PDF, если он был                                    |
| REQ-05 | История измерений, отсортированная по дате DESC                                                            |
| REQ-06 | Удаление одного измерения с подтверждением                                                                 |
| REQ-07 | Запрет редактирования сохранённых измерений (только удалить и создать заново)                              |
| REQ-08 | Авторасчёт BMI = weight_kg / (height_m^2), если рост известен из профиля                                    |
| REQ-09 | Уведомление при отсутствии нового измерения >30 дней (см. [011](011-notifications.md))                     |
| REQ-10 | Все измерения доступны как train/eval data для AI-блоков ([008](008-ai-inbody-predictor.md))               |

## 5. Non-functional Requirements

| #      | Requirement                                                          | Category    |
|--------|----------------------------------------------------------------------|-------------|
| NFR-01 | InBody-данные шифруются at-rest (медицински-чувствительные)          | Security    |
| NFR-02 | Парсинг PDF ≤5 секунд на 95-перцентиле (см. [013](013-inbody-pdf-import.md)) | Performance |
| NFR-03 | Список истории до 100 записей открывается ≤500 ms                    | Performance |
| NFR-04 | Оригинальный PDF доступен только владельцу (signed URL, TTL 5 минут) | Security    |

## 6. Data Model

**Entity: InBodyMeasurement**

| Field                | Type      | Constraints                         | Description                                          |
|----------------------|-----------|-------------------------------------|------------------------------------------------------|
| id                   | UUID      | PK                                  |                                                      |
| user_id              | UUID      | FK → User, required                 |                                                      |
| measured_at          | Timestamp | Required, ≤now                      | Когда было сделано измерение                         |
| weight_kg            | Number    | Required, 30..300                   |                                                      |
| height_cm            | Number    | Required, 100..250                  | Снапшот роста на момент измерения                    |
| sex                  | Enum      | Required, male/female               | Снапшот пола                                         |
| muscle_mass_kg       | Number    | Optional, 5..120                    |                                                      |
| body_fat_percent     | Number    | Required, 1..70                     |                                                      |
| body_water_percent   | Number    | Optional, 30..80                    |                                                      |
| protein_kg           | Number    | Optional, 1..30                     |                                                      |
| minerals_kg          | Number    | Optional, 0.5..10                   |                                                      |
| visceral_fat_level   | Integer   | Optional, 1..30                     |                                                      |
| bmr_kcal             | Number    | Optional, 500..5000                 | Может быть пустым → вычисляется из других полей      |
| fat_free_mass_kg     | Number    | Optional, 20..200                   |                                                      |
| bmi                  | Number    | Computed                            | weight_kg / (height_cm/100)^2                        |
| source               | Enum      | Required, `pdf` \| `manual`         |                                                      |
| original_pdf_url     | String    | Optional                            | Только если source=pdf                               |
| created_at           | Timestamp | Required, immutable                 |                                                      |

Эти поля соответствуют полям `in_body` в исходном ТЗ.

## 7. Screens

### Screen: InBody (главный)

**Purpose**: Свести историю измерений и точку входа для добавления нового.

**Elements**:
- Карточка «Последнее измерение» (большой счётчик: вес, % жира, мышечная масса)
- Сравнение с предыдущим (стрелки ↑/↓ + дельта)
- Кнопка «Добавить измерение» → экран выбора способа
- Список «Все измерения» (последние 5, кнопка «Показать все»)
- Если >30 дней — баннер «Пора пройти InBody снова»

**States**:
- Empty: «У вас ещё нет измерений» + CTA
- Loading, Error

### Screen: Добавить измерение (выбор способа)

**Elements**:
- Карточка «Загрузить PDF»
- Карточка «Ввести вручную»

### Screen: Ввод вручную

**Elements**: форма с обязательными и опциональными полями (см. data model).

### Screen: Превью PDF

**Elements**:
- Превью документа
- Распознанные поля с возможностью редактировать
- Кнопки «Сохранить» / «Отменить»

### Screen: Карточка измерения

**Elements**: все поля, источник, кнопки «Скачать PDF» / «Удалить».

## 9. API Endpoints

### Создать измерение (manual)

```
POST /api/v1/inbody/measurements
```

**Request**:
```json
{
  "measured_at": "2026-04-28T10:00:00Z",
  "weight_kg": 78.4,
  "body_fat_percent": 18.2,
  "muscle_mass_kg": 35.1,
  "visceral_fat_level": 7,
  "body_water_percent": 56.0,
  "bmr_kcal": 1750,
  "fat_free_mass_kg": 64.0
}
```

**Response 201**: полная запись с `id`, `bmi`, `source: "manual"`.

**Errors**:
| Code | Situation                          |
|------|------------------------------------|
| 400  | Невалидные значения / дата в будущем |
| 409  | Измерение на эту минуту уже существует |

### Загрузить PDF

```
POST /api/v1/inbody/measurements/from-pdf
```
multipart/form-data, поле `file`.

**Response 202**: `{"job_id": "...", "status": "parsing"}` — асинхронный flow.

```
GET /api/v1/inbody/measurements/from-pdf/{job_id}
```
**Response 200**:
```json
{
  "status": "ready",
  "extracted": { "weight_kg": 78.4, "body_fat_percent": 18.2, ... },
  "missing_fields": ["bmr_kcal"]
}
```

```
POST /api/v1/inbody/measurements/from-pdf/{job_id}/confirm
```
**Body**: финальные значения после правок пользователя.

### Список измерений

```
GET /api/v1/inbody/measurements?limit=20&offset=0
```
**Response 200**:
```json
{ "items": [...], "total": 47 }
```

### Получить одно

```
GET /api/v1/inbody/measurements/{id}
```

### Удалить

```
DELETE /api/v1/inbody/measurements/{id}
```

### Скачать оригинальный PDF

```
GET /api/v1/inbody/measurements/{id}/pdf
```
**Response 302**: redirect на signed URL (TTL 5 мин), либо 404 если source=manual.

## 10. Edge Cases

- Пользователь ввёл нереалистичные значения (вес 5 кг) → валидация по диапазонам.
- PDF битый/защищён паролем → ошибка парсинга, fallback на ручной ввод.
- Пользователь удалил все измерения → AI-блоки используют baseline из профиля.
- Два измерения в один и тот же день — допускается, если время отличается.
- Рост в момент измерения отличается от профиля (например, измерение из прошлого) — снапшот height_cm сохраняется.

---

## 11. Out of Scope

- Интеграция с весами/смарт-устройствами (Withings, Garmin) — отдельный эпик
- Импорт CSV пакетом — отдельная story
- Тренеры/третьи лица, добавляющие измерения за пользователя — B2B, отдельный эпик
- Редактирование сохранённого измерения (только удалить и создать заново)
