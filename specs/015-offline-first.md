# Specification: Offline-first архитектура клиента

**Epic:** E5 — Надёжность и UX в условиях нестабильной сети
**User Story:** Пользователь должен иметь возможность пользоваться приложением (просмотр и запись данных) без активного интернет-соединения, а сделанные изменения автоматически синхронизируются с сервером, как только связь восстановится.
**Related specs:** [003-inbody-measurements.md](003-inbody-measurements.md), [005-workout-tracking.md](005-workout-tracking.md), [006-ai-workout-generator.md](006-ai-workout-generator.md), [009-plan-adaptation.md](009-plan-adaptation.md), [013-inbody-pdf-import.md](013-inbody-pdf-import.md)

## 1. User Goal

Главные сценарии, в которых пользователь нуждается в offline-first:

1. **Тренажёрный зал без покрытия мобильной сети.** Самый частый случай — подвал спорт-комплекса, где сотовая сеть пропадает или 4G даёт <50 кбит/с. Пользователь должен иметь возможность: открыть план на сегодня, начать тренировку, фиксировать подходы, завершить — без единого сетевого запроса.
2. **Метро / самолёт / поездка.** Просмотр своей истории тренировок, последнего прогноза InBody, статистики — без интернета.
3. **Кратковременная потеря сети при отправке формы.** Пользователь жмёт «Сохранить замер InBody», в этот момент wifi моргнул — изменение не должно потеряться; нужно ретраить в фоне.

Не-цели:
- Real-time multi-device sync (изменения с другого устройства появляются мгновенно). Это отдельный спек.
- Полностью автономная работа без первого онбординга на онлайне (нужны exercise catalog, последний план, профиль).
- Offline-генерация плана (composer работает только на сервере).
- Offline-прогноз (ML-модель работает только на сервере; offline отдаём последний кэшированный ответ).
- Offline-парсинг InBody PDF (pdfplumber серверный).

## 2. Context

Архитектурное решение — **локальная БД на клиенте + sync-очередь**:

- На клиенте (Flutter PWA + Android) появляется локальная БД, зеркалирующая нужные таблицы прод-БД: профиль, история InBody, активный план, тренировки, exercise-логи, последний прогноз, exercise-каталог.
- Все чтения UI идут из локальной БД (Repository pattern). API-запросы запускаются в фоне для обновления локальной копии.
- Все записи идут сначала в локальную БД + sync-очередь, UI обновляется немедленно (optimistic). Sync-воркер периодически пытается отправить операции на сервер.
- Идемпотентность операций обеспечивается **client-side UUID** (`client_id`) — сервер дедуплицирует по нему.
- Конфликты разрешаются по стратегии **server-wins для read-only данных** (план, каталог, прогноз) и **last-write-wins по `updated_at`** для мутируемых сущностей (профиль). Для append-only сущностей (workouts, exercise_logs, inbody_measurements) конфликт невозможен — записи только добавляются, дедупликация по `client_id`.

Хранилище:
- **Android**: SQLite через `drift` (или `sqflite`).
- **Web (PWA)**: IndexedDB через `drift_web` или `sqflite_common_ffi_web`. Опция fallback на in-memory — для приватных сессий.
- Один декларативный schema в Dart, кросс-платформенный.

## 3. User Scenarios

### Scenario 1 — Тренировка в зале без сети

Пользователь приходит в зал, открывает приложение. Перед началом тренировки телефон теряет 4G (вошли в подвал).

**How to test independently**: Devtools «Throttling: Offline» в Chrome (или Airplane Mode на Android). Открыть приложение → /home → /training → активная тренировка → лог 5 подходов → finish. Все экраны рендерятся, все операции принимаются.

**Acceptance criteria**:

1. **Given** активный план есть в локальной БД (синхронизирован при последнем онлайне).
   **When** пользователь открывает «План» без сети.
   **Then** план открывается мгновенно, экран не показывает спиннер дольше 200 мс, нет сообщений об ошибке.

2. **Given** пользователь нажал «Начать тренировку».
   **When** клиент не имеет сети.
   **Then** новый workout создаётся в локальной БД с `client_id = uuid()`, `status = 'in_progress'`, помещается в sync-очередь как pending. UI сразу открывает экран активной тренировки.

3. **Given** пользователь записывает 12 подходов в течение часа.
   **When** все 12 уходят в локальную БД.
   **Then** каждый подход синхронно появляется в UI (без флика), 12 ops появляются в sync-очереди со статусом `pending`.

4. **Given** пользователь жмёт «Завершить» без сети.
   **When** клиент офлайн.
   **Then** finish записывается локально, UI показывает summary, в правом верхнем углу — индикатор «N изменений не отправлены».

### Scenario 2 — Восстановление сети, автосинхронизация

Пользователь выходит из зала, возвращается мобильная сеть.

**Acceptance criteria**:

1. **Given** в sync-очереди ≥1 pending-операция, клиент только что получил сетевой connectivity-event.
   **When** ConnectivityListener фиксирует online.
   **Then** SyncWorker запускается, отправляет операции по очереди (FIFO). Каждая операция помечается `applied` при HTTP 2xx или остаётся pending при сетевой ошибке.

2. **Given** все ops синхронизировались.
   **When** очередь опустошается.
   **Then** индикатор «N изменений не отправлены» исчезает, fallback-таймер до следующей попытки сбрасывается, локальная БД обновляется server-side ID'шниками (если они отличаются от client_id).

3. **Given** сервер ответил 409 Conflict (например, превышен лимит планов в active).
   **When** SyncWorker получает не-2xx.
   **Then** операция помечается `failed` с error-payload, пользователь видит баннер «Не удалось синхронизировать тренировку: …» с CTA «Повторить».

### Scenario 3 — Чтение данных с graceful degradation

Пользователь открывает «Прогноз» в офлайне.

**Acceptance criteria**:

1. **Given** последний прогноз InBody закэширован в локальной БД (запрашивался <24ч назад).
   **When** пользователь открывает экран прогноза в офлайне.
   **Then** показывается закэшированный прогноз с подписью «Последнее обновление: 14:32». Кнопка «Пересчитать» неактивна, при тапе — снэкбар «Нет сети».

2. **Given** прогноза в кэше нет (cold start без предыдущего онлайн-открытия экрана).
   **When** пользователь открывает экран в офлайне.
   **Then** empty-state: «Прогноз появится, как только подключитесь к сети». Без техно-stacktrace.

### Scenario 4 — Дедупликация при повторной отправке

Сеть пропала после того, как клиент отправил POST `/workouts`, но до получения ответа.

**Acceptance criteria**:

1. **Given** POST `/workouts` с `client_id=X` уже долетел до сервера, ответ не дошёл до клиента; клиент ретрайт через 30 сек.
   **When** retry приходит на сервер с тем же `client_id=X`.
   **Then** сервер находит существующий workout по `client_id`, возвращает 200 OK с тем же server-side `id`. Клиент обновляет локальный mapping.

2. **Given** операция выполнилась успешно, но retry уже отправлен.
   **When** сервер не нашёл существующий workout с `client_id=X` (race condition).
   **Then** клиент трактует второе 200 OK как успешный sync; локальная запись остаётся одна (deduped по client_id).

### Scenario 5 — Конфликт при многократном редактировании профиля

Пользователь редактирует профиль на устройстве A в офлайне (поменял goal). Параллельно на устройстве B (другое мобильное) поменял goal иначе. Оба устройства синкаются.

**Acceptance criteria**:

1. **Given** на сервере профиль с `updated_at = T1`, на устройстве A pending-операция с `client_updated_at = T2 > T1`.
   **When** A синкается, отправляет PATCH с заголовком `If-Unmodified-Since: T1`.
   **Then** сервер принимает (T1 совпадает), профиль становится с goal от A, `updated_at = T2`.

2. **Given** позже на устройстве B аналогичная pending-операция с `client_updated_at = T3` (где T3 локально новее, но сервер уже сдвинулся к T2).
   **When** B отправляет PATCH с `If-Unmodified-Since: T1` (стороной A).
   **Then** сервер возвращает 412 Precondition Failed. Клиент B показывает баннер «Профиль был изменён на другом устройстве. Принять серверную версию или отправить свои изменения?». Пользователь выбирает.

### Scenario 6 — Полный rebuild локальной БД

Пользователь переустановил приложение или вышел/зашёл.

**Acceptance criteria**:

1. **Given** локальная БД пуста после logout.
   **When** пользователь логинится.
   **Then** клиент дозагружает: профиль, активный план, последние 50 тренировок, последние 12 InBody, последний прогноз, exercise catalog (~400 шт). Прогресс-бар «Загрузка ваших данных… N/5».

2. **Given** download прерван (потеря сети между профилем и планом).
   **When** запускается ConnectivityListener после восстановления.
   **Then** download продолжается с того места, где упал (по checkpoint'ам), без дублей.

## 4. Functional Requirements

- **REQ-01.** Каждая мутирующая операция клиента (POST/PATCH/DELETE) проходит через sync-очередь и имеет уникальный `client_id` (UUID v4).
- **REQ-02.** Сервер дедуплицирует операции по `client_id` (новая колонка `client_id TEXT UNIQUE` в `workouts`, `exercise_logs`, `inbody_measurements`).
- **REQ-03.** Все чтения экранов (List, Detail) идут сначала из локальной БД с эмиссией данных через Stream<T>, затем фоном обновляются через API. UI всегда что-то показывает, пока есть кэш.
- **REQ-04.** ConnectivityListener (`connectivity_plus`) триггерит SyncWorker при переходе offline → online.
- **REQ-05.** Sync-очередь обрабатывается FIFO. Failed-операции ретрайтся с exponential backoff (1с → 2с → 4с → … capped at 30с).
- **REQ-06.** Для мутируемых сущностей (профиль) клиент шлёт `If-Unmodified-Since: <last_known_updated_at>`. Сервер возвращает 412 при расхождении.
- **REQ-07.** Append-only сущности (workout, exercise_log, inbody) НЕ требуют конфликт-резолва — сервер просто инсёртит при отсутствии `client_id` или возвращает 200 с существующей записью.
- **REQ-08.** Read-only данные (план, прогноз, каталог) обновляются по ETag/If-None-Match — `304 Not Modified` экономит трафик.
- **REQ-09.** Локальная БД эвиктит самые старые тренировки/логи, когда суммарный размер превышает 50 MB. Сохраняется минимум: профиль + активный план + 30 последних тренировок + 6 последних InBody + последний прогноз + exercise catalog.
- **REQ-10.** Бэйдж pending-операций виден в навигационной шапке любого экрана.

## 5. Non-functional Requirements

- **NFR-01.** Любой просмотровый экран (Home, Plan, History, Profile) открывается из локальной БД за <300 мс.
- **NFR-02.** Любая запись (новый подход, новый замер, изменение профиля) принимается локально за <100 мс.
- **NFR-03.** Sync-очередь не теряет операции при крэше / kill процесса (Android) или закрытии вкладки (Web). Persisted в той же локальной БД.
- **NFR-04.** Размер локальной БД на клиенте — медианно ≤30 MB; жёсткий cap 80 MB (после превышения — эвикция).
- **NFR-05.** При первом запуске после установки — initial sync укладывается в 10 секунд на 3G (~750 КБ overhead).
- **NFR-06.** UI остаётся отзывчивым во время initial sync (download идёт в isolate / worker).

## 6. Data Model

### Локальная БД клиента (новая схема, drift)

```
clients_state — singleton-таблица с метаданными последнего sync
  user_id, last_synced_at, sync_version

profiles — зеркало user_profiles с добавлением:
  + dirty BOOL — есть локальные несинхронизированные изменения
  + client_updated_at TIMESTAMPTZ — для If-Unmodified-Since

workouts — зеркало:
  + client_id UUID UNIQUE
  + sync_status ENUM('synced','pending','failed')

exercise_logs — зеркало:
  + client_id UUID UNIQUE
  + sync_status

inbody_measurements — зеркало:
  + client_id UUID UNIQUE
  + sync_status

forecasts — кэш последнего прогноза с TTL:
  + generated_at, ttl_seconds=86400

plans — зеркало активного плана. read-only с клиента (генерирует сервер).

exercises — кэш каталога:
  + version INT — для инвалидации (если сервер выкатил новый список)

sync_queue — pending-операции:
  id, client_id (FK), op_type ('CREATE'|'UPDATE'|'DELETE'),
  endpoint, payload JSONB, attempts INT, last_error TEXT,
  next_retry_at TIMESTAMPTZ, status ENUM('pending','in_flight','applied','failed')
```

### Изменения в серверной БД

```sql
ALTER TABLE workouts ADD COLUMN client_id UUID UNIQUE;
ALTER TABLE exercise_logs ADD COLUMN client_id UUID UNIQUE;
ALTER TABLE inbody_measurements ADD COLUMN client_id UUID UNIQUE;
CREATE INDEX ix_workouts_client_id ON workouts(client_id) WHERE client_id IS NOT NULL;
-- аналогично для exercise_logs и inbody_measurements
```

Миграция `0021_offline_client_ids.py`. Backfill не нужен (NULL для старых).

## 8. API Contracts

### Заголовки запросов

- `X-Client-Op-Id: <uuid>` — идентификатор pending-операции для логов сервера и dedup.
- `If-Unmodified-Since: <iso>` — только на PATCH /profile (и других мутируемых сущностях с last-write-wins).
- `If-None-Match: <etag>` — на GET плана/прогноза/каталога.

### Эндпоинты — изменения

- `POST /workouts` — принимает поле `client_id` в body. Если запись с этим client_id уже существует — возвращает 200 OK с существующим объектом (idempotent).
- `POST /workouts/{id}/logs` — поле `client_id` в body. Аналогично.
- `POST /inbody/measurements` — поле `client_id`. Аналогично.
- `PATCH /profile` — поддерживает `If-Unmodified-Since`. При расхождении — 412.
- `GET /catalog/exercises?since=<version>` — incremental sync каталога; возвращает только diff с указанной версии (если поддерживается).

### Новый эндпоинт (опционально)

- `GET /sync/snapshot` — собирает в одном вызове: профиль + активный план + последние N тренировок + последние M InBody + последний прогноз + версию каталога. Для initial sync после login, экономит N round-trip'ов.

## 9. Edge Cases

- **Чистка локальной БД на стороне браузера** (пользователь нажал «Clear site data» в Chrome). Клиент видит пустую БД при наличии валидного refresh-token → запускает initial sync.
- **Sync во время активной тренировки.** Не блокировать UI. Sync крутится в isolate (Android) / WebWorker (Web) и не трогает экран лога.
- **Авто-logout по идемпотентности.** Если pending-операция получает 401 (token истёк) — refresh-token rotation; если refresh тоже failed → mark очередь как `blocked_auth`, выкидывает на login, при следующем успешном login очередь восстанавливается (если тот же user_id).
- **Конфликт client_id (теоретический).** UUID v4 коллизия немыслима (1 на 2^61). Сервер при дубле возвращает 409 → клиент локально перевыпускает client_id и ретрайт. В практике это код от багов клиента, не от коллизий.
- **Размер очереди вырос > 1000 ops** (пользователь месяц без сети). Покажем баннер «Слишком много несохранённых изменений, проверьте интернет», ограничим запись новых.
- **DELETE-операции для уже несинхронизированных CREATE.** Если в очереди есть CREATE с client_id=X, и пользователь делает DELETE того же объекта — обе операции склеиваются в no-op (cancel).
- **Изменение часового пояса устройства.** Все timestamp'ы в UTC; на клиенте storage в UTC, UI рендерит в локальной TZ.

## 10. Success Criteria

- **SC-01.** Cold-launch главного экрана в офлайне = <300 мс (от тапа по иконке до отрисовки).
- **SC-02.** Полная тренировка (5 упражнений × 3 подхода = 15 sets + finish) офлайн отрабатывает без single network call; все 17 операций успешно синхронизируются после восстановления сети.
- **SC-03.** Прерванный sync (kill процесса посреди отправки 12-й из 15 операций) при следующем запуске продолжает с 12-й, без дублей в БД.
- **SC-04.** Размер локальной БД активного пользователя через 6 месяцев использования — <30 MB.
- **SC-05.** Нет крэшей / необработанных исключений на 100 manual-тестах в режиме offline.

## 11. Open Questions

- **Q1.** Стоит ли поддерживать «глубокий» offline для прогноза — например, простой клиентский Ridge fallback (loaded with weights at build-time)? Вряд ли, отдельная спека.
- **Q2.** GraphQL / WebSocket subscriptions для real-time multi-device sync — отдельный epic. Сейчас стратегия — pull, не push.
- **Q3.** Encryption-at-rest для локальной БД: использовать SQLCipher на Android? На Web — что? IndexedDB шифровать нечем стандартным. Если хранятся health-данные (InBody) — это **может быть** критично для совместимости с ФЗ-152. Эту спеку нужно обсудить с юристом.
- **Q4.** При выходе пользователя из аккаунта — стираем локальную БД или оставляем (на случай, если он залогинится снова под тем же user_id)? Простое решение: стирать всегда. Сложное: хранить под `user_id`-партицией.
