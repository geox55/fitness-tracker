# Specification: Учёт тренировок

**Epic:** E2 — Тренировки
**User Story:** Пользователь должен фиксировать каждую тренировку — какие упражнения сделал, сколько подходов и повторений, с каким весом — чтобы видеть прогресс и кормить им AI-блоки.
**Related specs:** [004-exercise-catalog.md](004-exercise-catalog.md), [006-ai-workout-generator.md](006-ai-workout-generator.md), [008-ai-inbody-predictor.md](008-ai-inbody-predictor.md), [010-analytics-reports.md](010-analytics-reports.md)

---

## 1. User Goal

Пользователь идёт в зал по плану, который сгенерировал AI ([006](006-ai-workout-generator.md)). На каждое упражнение он фиксирует, сколько реально сделал подходов, повторений, с каким весом, и при желании — RPE и время отдыха. Если он отклоняется от плана (заменил упражнение, пропустил, добавил своё) — это тоже сохраняется. После тренировки он видит саммари и попадает в историю.

История тренировок — основа для:
- графиков прогресса ([010](010-analytics-reports.md))
- адаптации плана ([009](009-plan-adaptation-and-chat.md))
- ML-предиктора InBody ([008](008-ai-inbody-predictor.md))

---

## 2. Context

Каждая тренировка — это коллекция exercise_logs (подходов). Поля соответствуют полям из исходного ТЗ: `workouts(workout_id, user_id, performed_at)`, `exercise_logs(workout_id, exercise_id, set_number, reps, weight_kg, rpe, rest_seconds)`.

Тренировка может быть запланирована (originated from AI plan), или ad-hoc (пользователь начал «свободную» тренировку). После завершения структура одинаковая.

## 3. User Scenarios

### Scenario 1 — Старт тренировки по плану

Сегодня по плану — день А (грудь/трицепс). Пользователь нажимает «Начать тренировку» и видит список упражнений с целевыми подходами/повторениями.

**How to test independently**: Сгенерировать план (см. [006](006-ai-workout-generator.md)), на главном экране нажать «Начать сегодняшнюю тренировку», убедиться что подгрузился именно план дня.

**Acceptance criteria**:

1. **Given** у пользователя есть активный план и сегодня — день тренировки.
   **When** он нажимает «Начать тренировку».
   **Then** создаётся запись Workout со статусом `in_progress`, открывается экран тренировки с предзагруженными упражнениями (из плана) и пустыми ячейками подходов.

2. **Given** у пользователя нет плана.
   **When** он открывает главный экран.
   **Then** видит CTA «Сгенерировать план» (→ [006](006-ai-workout-generator.md)) или «Начать свободную тренировку».

### Scenario 2 — Фиксация подхода

Пользователь сделал подход — вводит вес и повторения.

**How to test independently**: Начать тренировку, ввести данные подхода, нажать «Сохранить подход», проверить что в истории появилась запись.

**Acceptance criteria**:

1. **Given** пользователь на экране активной тренировки.
   **When** он вводит вес (кг) и повторения для подхода 1, нажимает галочку.
   **Then** создаётся ExerciseLog с `set_number = 1`, `reps`, `weight_kg`, остальные поля nullable, статус подхода — «выполнен».

2. **Given** есть предыдущая тренировка с этим же упражнением.
   **When** пользователь начинает новый подход.
   **Then** в полях вес/повторения по умолчанию подставляются значения из последнего выполненного подхода (suggestion).

3. **Given** план рекомендует 4 подхода × 8 повторений с весом 60 кг.
   **When** пользователь сделал 4×8.
   **Then** упражнение помечается как полностью выполненное (зелёная галка).

4. **Given** пользователь сделал больше или меньше повторений/подходов, чем в плане.
   **When** он сохраняет.
   **Then** значения сохраняются как есть; в саммари тренировки помечается «выше/ниже плана».

### Scenario 3 — Замена/пропуск упражнения

Зал занят — пользователь хочет заменить упражнение или пропустить.

**Acceptance criteria**:

1. **Given** упражнение в плане ещё не начато.
   **When** пользователь нажимает «Заменить» и выбирает другое из каталога ([004](004-exercise-catalog.md)).
   **Then** в текущей тренировке упражнение заменяется; план на день не пересобирается.

2. **Given** упражнение в плане ещё не начато.
   **When** пользователь нажимает «Пропустить».
   **Then** упражнение помечается `skipped = true`, продолжается тренировка со следующего.

### Scenario 4 — Свободная (ad-hoc) тренировка

Пользователь хочет «просто потренироваться» вне плана.

**Acceptance criteria**:

1. **Given** пользователь на главном экране.
   **When** он нажимает «Свободная тренировка».
   **Then** создаётся пустой Workout, пользователь сам добавляет упражнения из каталога и фиксирует подходы.

### Scenario 5 — Завершение тренировки

**Acceptance criteria**:

1. **Given** пользователь закончил все/часть упражнений.
   **When** он нажимает «Завершить тренировку».
   **Then** статус Workout меняется на `completed`, проставляется `finished_at`, показывается экран саммари.

2. **Given** саммари тренировки.
   **When** оно отображается.
   **Then** видно: длительность, число выполненных упражнений, общий тоннаж (сумма weight × reps), общее число подходов, кнопка «На главный».

3. **Given** пользователь забыл завершить тренировку.
   **When** прошло >12 часов с последней активности в Workout.
   **Then** Workout автоматически закрывается (`auto_finished = true`); пользователь получает уведомление.

### Scenario 6 — История тренировок

**Acceptance criteria**:

1. **Given** у пользователя есть N тренировок.
   **When** он открывает «История».
   **Then** видит список по `performed_at` DESC: дата, длительность, тоннаж, количество упражнений, бейдж «по плану» / «свободная».

2. **Given** пользователь тапает на тренировку.
   **When** открывается детальная карточка.
   **Then** видит все упражнения с подходами, может удалить тренировку.

### Scenario 7 — Удаление тренировки

**Acceptance criteria**:

1. **Given** пользователь открыл тренировку из истории.
   **When** он нажимает «Удалить» и подтверждает.
   **Then** Workout и все её ExerciseLog удаляются (hard delete); графики обновляются.

## 4. Functional Requirements

| #      | Requirement                                                                                                              |
|--------|--------------------------------------------------------------------------------------------------------------------------|
| REQ-01 | Старт тренировки по сегодняшнему плану (если есть)                                                                       |
| REQ-02 | Старт свободной тренировки (без плана)                                                                                   |
| REQ-03 | Добавление упражнения в тренировку из каталога ([004](004-exercise-catalog.md))                                          |
| REQ-04 | Фиксация подхода: weight_kg, reps; опционально rpe (1..10), rest_seconds                                                  |
| REQ-05 | Auto-suggestion веса/повторений из последнего выполненного подхода для того же упражнения                                |
| REQ-06 | Замена упражнения внутри активной тренировки                                                                             |
| REQ-07 | Пропуск упражнения внутри активной тренировки                                                                            |
| REQ-08 | Завершение тренировки с расчётом саммари (тоннаж, длительность, число подходов)                                          |
| REQ-09 | Авто-завершение тренировки через 12 часов простоя                                                                        |
| REQ-10 | История тренировок с фильтром по дате и типу (по плану / свободная)                                                      |
| REQ-11 | Удаление тренировки                                                                                                      |
| REQ-12 | Связь с планом тренировок (поле `plan_day_id` или `plan_id`) для AI-адаптации                                            |
| REQ-13 | Локальное сохранение прогресса тренировки на клиенте — на случай потери сети и закрытия приложения                       |

## 5. Non-functional Requirements

| #      | Requirement                                                                | Category    |
|--------|----------------------------------------------------------------------------|-------------|
| NFR-01 | Сохранение подхода ≤200 ms на 95-перцентиле (UX critical в зале)           | Performance |

## 6. Data Model

**Entity: Workout**

| Field          | Type      | Constraints                                                | Description                              |
|----------------|-----------|------------------------------------------------------------|------------------------------------------|
| workout_id     | UUID      | PK                                                         | Совпадает с полем из ТЗ                  |
| user_id        | UUID      | FK → User, required                                        |                                          |
| performed_at   | Timestamp | Required                                                   | Когда тренировка началась                |
| finished_at    | Timestamp | Optional                                                   | Null = ещё не завершена                  |
| status         | Enum      | `in_progress` \| `completed` \| `auto_finished` \| `cancelled` |                                      |
| origin         | Enum      | `plan` \| `freestyle`                                      |                                          |
| plan_day_id    | UUID      | Optional, FK → PlanDay (если origin=plan)                  | См. [006](006-ai-workout-generator.md)   |
| notes          | Text      | Optional, ≤500 chars                                       |                                          |

**Entity: ExerciseLog**

| Field         | Type     | Constraints                                  | Description                                       |
|---------------|----------|----------------------------------------------|---------------------------------------------------|
| id            | UUID     | PK                                           |                                                   |
| workout_id    | UUID     | FK → Workout, required                       |                                                   |
| exercise_id   | UUID     | FK → Exercise, required                      |                                                   |
| set_number    | Integer  | Required, ≥1                                 |                                                   |
| reps          | Integer  | Required, 1..200                             |                                                   |
| weight_kg     | Number   | Required for non-bodyweight, 0..500          | 0 для bodyweight                                  |
| rpe           | Integer  | Optional, 1..10                              | Rate of Perceived Exertion                        |
| rest_seconds  | Integer  | Optional, 0..1800                            |                                                   |
| skipped       | Boolean  | Default false                                | Если true — это запланированный, но пропущенный   |
| logged_at     | Timestamp| Required                                     |                                                   |

Эти поля совпадают со списком из исходного ТЗ.

## 7. Screens

### Screen: Активная тренировка

**Purpose**: Фиксация подходов в реальном времени.

**Elements**:
- Таймер тренировки (длительность с момента старта)
- Список упражнений с текущим подсвеченным
- Для текущего: название, ссылка на технику, целевые подходы/повторения, форма ввода (weight, reps, rpe, rest)
- Таймер отдыха между подходами (опц.)
- Кнопки: «Заменить», «Пропустить», «Завершить»

**States**:
- Loading при сохранении подхода
- Offline indicator если без сети + локальный кэш
- Error при синхронизации → retry

### Screen: Саммари тренировки

**Elements**: длительность, тоннаж, число подходов, список упражнений с резюме, кнопка «На главный».

### Screen: История тренировок

**Elements**: список с фильтрами по дате и типу, infinite scroll, тап на запись → детальная карточка.

### Screen: Карточка тренировки (read-only)

**Elements**: то же саммари + полный список ExerciseLog, кнопка «Удалить».

## 9. API Endpoints

### Старт тренировки

```
POST /api/v1/workouts
```
**Request**:
```json
{ "origin": "plan", "plan_day_id": "uuid" }
```
или
```json
{ "origin": "freestyle" }
```

**Response 201**: полный Workout со статусом `in_progress`.

### Завершить тренировку

```
POST /api/v1/workouts/{workout_id}/finish
```
**Response 200**: Workout с `status=completed`, `finished_at`.

### Отменить

```
POST /api/v1/workouts/{workout_id}/cancel
```

### Записать подход

```
POST /api/v1/workouts/{workout_id}/logs
```
**Request**:
```json
{
  "exercise_id": "uuid",
  "set_number": 2,
  "reps": 8,
  "weight_kg": 60,
  "rpe": 8,
  "rest_seconds": 90
}
```
**Response 201**: ExerciseLog.

**Errors**: 409 если workout не `in_progress`.

### Замена упражнения

```
PATCH /api/v1/workouts/{workout_id}/exercises/{old_exercise_id}/replace
{ "new_exercise_id": "uuid" }
```

### Пропуск упражнения

```
POST /api/v1/workouts/{workout_id}/exercises/{exercise_id}/skip
```

### История

```
GET /api/v1/workouts?from=2026-01-01&to=2026-04-30&origin=plan&limit=20&offset=0
```

### Получить одну

```
GET /api/v1/workouts/{workout_id}
```

### Удалить

```
DELETE /api/v1/workouts/{workout_id}
```

## 10. Edge Cases

- Сеть пропала во время тренировки → данные пишутся локально и синхронизируются позже; пользователь видит индикатор «офлайн».
- Пользователь начал две тренировки в разные дни без завершения → старая авто-закрывается через 12 ч, новая продолжается.
- Bodyweight упражнения → `weight_kg = 0`, валидация не падает.
- Пользователь редактирует уже сохранённый подход → разрешено в течение текущей сессии тренировки (до finish), после — read-only.
- Удаление упражнения из каталога администратором → существующие логи остаются, ссылка показывается как «Удалённое упражнение».

## 11. Out of Scope

- Социальный шаринг тренировок
- Тренировки в реальном времени с тренером
- Wearable-интеграции (Apple Watch ритм/калории) — отдельный эпик
- Распознавание упражнений по видео
- Расчёт 1RM (one-rep max) — может появиться в [010](010-analytics-reports.md) фаза 2
