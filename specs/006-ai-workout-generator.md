# Specification: AI генератор плана тренировок (своя ML-модель)

**Epic:** E3 — AI-блок Егора
**User Story:** Пользователь должен получать персонализированный план тренировок на 4 недели, сформированный собственной ML-моделью на основе InBody, цели, уровня подготовки и доступного оборудования.
**Related specs:** [002-user-profile.md](002-user-profile.md), [003-inbody-measurements.md](003-inbody-measurements.md), [004-exercise-catalog.md](004-exercise-catalog.md), [005-workout-tracking.md](005-workout-tracking.md), [009-plan-adaptation-and-chat.md](009-plan-adaptation-and-chat.md), [012-ml-dataset.md](012-ml-dataset.md)

---

## 1. User Goal

Пользователь только что заполнил профиль и добавил InBody. Он не хочет искать программы в интернете и подбирать упражнения сам — он хочет один раз нажать «Сгенерировать план» и получить готовую программу на 4 недели с разбивкой по дням, упражнениями, подходами/повторениями и рекомендациями по прогрессии. План должен учитывать его цель (похудение / набор массы / поддержание формы), уровень подготовки, частоту тренировок и доступное оборудование.

Это **главный AI-блок Егора** для диплома: своя ML-модель, обученная на объединённом датасете ([012](012-ml-dataset.md)).

---

## 2. Context

Архитектура модели — owned by Егор и описывается в магистерской главе. С точки зрения продукта здесь фиксируется:
- Что подаётся на вход (фичи).
- Что должно вернуться (структура плана).
- Какие правила/ограничения должны соблюдаться (баланс групп мышц, частота, подбор по оборудованию).
- Как пользователь взаимодействует с генерацией.

ML-модель работает в гибридной схеме:
1. **Recommender** (обучаемая часть) — ранжирует упражнения по релевантности фичам пользователя.
2. **Rule-based composer** (детерминированная часть) — собирает выбранные упражнения в недельный сплит по правилам (например, не нагружать одну группу два дня подряд, balance push/pull, прогрессия объёма).

Подробности обучения, метрик и архитектуры → [012](012-ml-dataset.md) и магистерская работа Егора.

## 3. User Scenarios

### Scenario 1 — Первая генерация плана

Пользователь только что закончил онбординг и впервые добавил InBody.

**How to test independently**: Создать тестового пользователя со всеми обязательными полями + 1 InBody, нажать «Сгенерировать план», проверить структуру результата.

**Acceptance criteria**:

1. **Given** профиль завершён + есть ≥1 InBody-измерение.
   **When** пользователь нажимает «Сгенерировать план».
   **Then** ML-модель формирует план: 4 недели × N тренировочных дней (N = `training_frequency` из профиля), каждая тренировка содержит 4–8 упражнений с указанием подходов/повторений/целевого RPE/отдыха.

2. **Given** генерация может занять до 10 секунд.
   **When** запрос отправлен.
   **Then** показывается экран загрузки с прогресс-индикатором; в случае timeout (>30 сек) — ошибка с retry.

3. **Given** план сгенерирован.
   **When** пользователь его видит.
   **Then** есть итог: цель, частота, длительность, общий тоннаж по неделям; и разбивка по неделям → дням → упражнениям.

### Scenario 2 — Учёт оборудования

Пользователь занимается дома (только гантели + турник).

**How to test independently**: В профиле выставить `equipment_available = [dumbbell, bodyweight, pullup_bar]`, сгенерировать план, убедиться что ни одно упражнение не требует другого оборудования.

**Acceptance criteria**:

1. **Given** в профиле указан ограниченный набор оборудования.
   **When** план генерируется.
   **Then** все упражнения в плане удовлетворяют `equipment ⊆ equipment_available`.

2. **Given** для пользователя нельзя составить полноценный план в рамках ограничения (например, только bodyweight и цель «набор массы»).
   **When** ML-модель не может предложить минимум упражнений на группу мышц.
   **Then** возвращается план с пометкой «ограниченный набор» + warning «для лучших результатов добавьте: гантели, штангу».

### Scenario 3 — Прогрессия нагрузки

План на 4 недели должен включать прогрессию.

**How to test independently**: Сгенерировать план, проверить что веса/повторения для одного и того же упражнения растут от недели 1 к 4.

**Acceptance criteria**:

1. **Given** план на 4 недели с одним и тем же упражнением.
   **When** просматривается прогрессия.
   **Then** в большинстве упражнений недели 2–4 имеют либо +2.5..5 кг, либо +1..2 повторения, либо +1 подход относительно предыдущей недели (Linear Progression / Double Progression).

2. **Given** уровень `beginner`.
   **When** план генерируется.
   **Then** прогрессия Linear (фиксированный шаг каждую неделю), упражнения базовые (squat, deadlift, bench, row, OHP, pullup и их регрессии).

3. **Given** уровень `advanced`.
   **When** план генерируется.
   **Then** возможна периодизация (volume → intensity), допускаются изоляции и сложные движения.

### Scenario 4 — Кардио-рекомендации

В зависимости от цели пользователя план содержит блок кардио.

**Acceptance criteria**:

1. **Given** цель `weight_loss`.
   **When** план сгенерирован.
   **Then** добавляется кардио: 2–3 сессии/неделю, длительность 25–45 мин, тип (LISS/HIIT) с пометкой «опционально».

2. **Given** цель `muscle_gain`.
   **When** план сгенерирован.
   **Then** кардио минимально (1 LISS/неделю, ≤30 мин) или отсутствует.

3. **Given** цель `maintenance`.
   **When** план сгенерирован.
   **Then** кардио 1–2 сессии/неделю.

### Scenario 5 — Просмотр плана

Пользователь хочет посмотреть план целиком и по дням.

**Acceptance criteria**:

1. **Given** план активен.
   **When** пользователь открывает «Мой план».
   **Then** видит обзор: 4 недели в виде вкладок, в каждой — дни с упражнениями.

2. **Given** на сегодня запланирована тренировка.
   **When** пользователь открывает главный экран.
   **Then** видит карточку «Сегодня: День А — Грудь/Трицепс» с CTA «Начать» (см. [005](005-workout-tracking.md)).

### Scenario 6 — Ручная регенерация плана

**Acceptance criteria**:

1. **Given** пользователь смотрит активный план.
   **When** он нажимает «Сгенерировать заново» и подтверждает.
   **Then** старый план архивируется (status=archived), создаётся новый; история тренировок сохраняется.

### Scenario 7 — Автоматическая регенерация

См. [009-plan-adaptation-and-chat.md](009-plan-adaptation-and-chat.md). После 4 недель или при изменении условий — план перегенерируется автоматически.

## 4. Functional Requirements

| #      | Requirement                                                                                                                       |
|--------|-----------------------------------------------------------------------------------------------------------------------------------|
| REQ-01 | Генерация плана на 4 недели с разбивкой по дням                                                                                  |
| REQ-02 | Учёт InBody (последний снимок): вес, % жира, мышечная масса, BMR                                                                  |
| REQ-03 | Учёт цели (`weight_loss` \| `muscle_gain` \| `maintenance`)                                                                       |
| REQ-04 | Учёт уровня подготовки (`beginner` \| `intermediate` \| `advanced`)                                                                |
| REQ-05 | Учёт частоты тренировок (2–6 раз/неделю)                                                                                          |
| REQ-06 | Учёт доступного оборудования: только подмножество из `equipment_available`                                                        |
| REQ-07 | Каждая тренировка: 4–8 упражнений, разбивка по группам мышц без перегрузки одной группы два дня подряд                            |
| REQ-08 | На каждое упражнение: целевые подходы (`sets`), повторения (`reps`), целевой RPE, рекомендуемое время отдыха                       |
| REQ-09 | Прогрессия нагрузки: каждую неделю минимум +1 параметр (вес/повторения/подходы) для большинства упражнений                        |
| REQ-10 | Блок кардио, согласованный с целью                                                                                                |
| REQ-11 | Все выбранные упражнения — из каталога ([004](004-exercise-catalog.md))                                                           |
| REQ-12 | Возможность ручной регенерации плана                                                                                              |
| REQ-13 | Статус плана: `active` \| `archived`. Только один план может быть `active` на пользователя                                       |
| REQ-14 | API для возврата JSON-плана с детерминированной структурой (см. секцию 9)                                                          |
| REQ-15 | Логирование входных фичей и выхода модели для оффлайн-анализа метрик                                                              |
| REQ-16 | Fallback: если ML-модель упала / timeout — возврат rule-based плана из шаблонов; пользователь не видит ошибку                     |

## 5. Non-functional Requirements

| #      | Requirement                                                                                          | Category    |
|--------|------------------------------------------------------------------------------------------------------|-------------|
| NFR-01 | Генерация плана ≤10 секунд на 95-перцентиле                                                          | Performance |
| NFR-02 | Воспроизводимость: при одном и том же seed + входе модель возвращает тот же план (для отладки)       | Reliability |
| NFR-03 | Inference-сервис изолирован (отдельный контейнер/процесс), не блокирует основной API                 | Architecture |
| NFR-04 | Логи фичей пользователя анонимизированы перед сохранением для retraining                              | Privacy     |

## 6. Success Criteria

| #     | Criterion                                                          | How to measure                                                |
|-------|--------------------------------------------------------------------|---------------------------------------------------------------|
| SC-01 | План соответствует балансу групп мышц                              | Статически: ≥80% сгенерированных планов проходят валидатор   |
| SC-02 | Пользователи завершают ≥60% запланированных тренировок             | Ratio completed / planned по выборке тренировок               |
| SC-03 | Прогрессия нагрузки ведёт к росту тоннажа от недели 1 к 4          | Avg total tonnage week 4 > week 1 на ≥10% по выборке         |
| SC-04 | <5% планов содержат упражнения вне `equipment_available`            | Линт сгенерированных планов                                   |

## 7. Data Model

**Entity: WorkoutPlan**

| Field          | Type      | Constraints                              | Description                                          |
|----------------|-----------|------------------------------------------|------------------------------------------------------|
| id             | UUID      | PK                                       |                                                      |
| user_id        | UUID      | FK → User, required                      |                                                      |
| status         | Enum      | `active` \| `archived`                   | Только один active                                   |
| generated_at   | Timestamp | Required, immutable                      |                                                      |
| valid_until    | Date      | Required                                 | generated_at + 4 недели                              |
| goal           | Enum      | Required                                 | Снапшот цели на момент генерации                     |
| training_level | Enum      | Required                                 |                                                      |
| frequency      | Integer   | Required, 2..6                           |                                                      |
| input_features | JSON      | Required                                 | Все фичи, поданные в модель (для воспроизводимости) |
| model_version  | String    | Required                                 | Версия ML-модели                                     |

**Entity: PlanWeek**

| Field    | Type    | Constraints                              | Description |
|----------|---------|------------------------------------------|-------------|
| id       | UUID    | PK                                       |             |
| plan_id  | UUID    | FK → WorkoutPlan, required               |             |
| week_no  | Integer | Required, 1..4                           |             |

**Entity: PlanDay**

| Field      | Type    | Constraints                                  | Description                              |

| day_no     | Integer | Required, 1..7                               | Логический день недели                   |
| name       | String  | Required                                     | «День А — Грудь/Трицепс», «Кардио»       |

| Field            | Type     | Constraints                              | Description                                         |

| day_id           | UUID     | FK → PlanDay, required                   |                                                     |

## 8. Screens

### Screen: Генерация плана

**Purpose**: Кнопка запуска + индикатор прогресса.

**Elements**:
- Резюме входных параметров (цель, уровень, частота, оборудование, последние InBody)
- Кнопка «Сгенерировать план»
- При запуске — спиннер с этапами: «анализирую данные» → «подбираю упражнения» → «формирую недели» → «готово»

### Screen: Мой план (обзор)

**Elements**:
- Шапка: цель, длительность, число тренировок/неделю
- Вкладки 4 недель
- Внутри недели — карточки дней (День А, День B, День отдыха)
- Тап на день → детализация
- Кнопка «Сгенерировать заново» в меню

### Screen: День плана

**Elements**:
- Список упражнений с подходами, повторениями, RPE, отдыхом
- Тап на упражнение → карточка упражнения ([004](004-exercise-catalog.md))
- Кнопка «Начать тренировку» (см. [005](005-workout-tracking.md))

## 9. API Endpoints

### Сгенерировать план

```
POST /api/v1/plans/generate
```

**Request** (опционально, иначе берутся текущие значения профиля):
```json
{
  "override": {
    "training_frequency": 4,
    "equipment_available": ["barbell", "dumbbell", "bench", "pullup_bar"]
  }
}
```

**Response 201**:
```json
{
  "plan_id": "uuid",
  "status": "active",
  "generated_at": "2026-04-28T12:00:00Z",
  "valid_until": "2026-05-26",
  "goal": "muscle_gain",
  "training_level": "intermediate",
  "frequency": 4,
  "model_version": "workout-gen-0.3.1",
  "weeks": [
    {
      "week_no": 1,
      "days": [
        {
          "day_no": 1,
          "name": "День А — Грудь/Трицепс",
          "type": "strength",
          "exercises": [
            {
              "exercise_id": "uuid",
              "exercise_name": "Barbell Bench Press",
              "order_no": 1,
              "target_sets": 4,
              "target_reps_min": 6,
              "target_reps_max": 8,
              "target_rpe": 8,
              "rest_seconds": 180,
              "target_weight_kg": 60.0
            }
          ]
        }
      ]
    }
  ],
  "warnings": []
}
```

**Errors**:
| Code | Situation                                                                       |
|------|---------------------------------------------------------------------------------|
| 400  | Профиль не завершён или нет InBody → `{"error": "preconditions_not_met", "missing": [...]}` |
| 504  | Timeout модели → fallback rule-based, но в `warnings` помечается                |

### Получить активный план

```
GET /api/v1/plans/active
```
**Response 200**: тот же объект; `404` если активного плана нет.

### Получить любой план

```
GET /api/v1/plans/{plan_id}
```

### Архивировать

```
POST /api/v1/plans/{plan_id}/archive
```

### История планов

```
GET /api/v1/plans?status=archived&limit=10
```

## 10. Edge Cases

- Нет InBody-измерений → AI использует baseline из профиля + дисклеймер «для лучшей точности добавьте InBody».
- Цель `weight_loss` + уровень `beginner` + частота 2 → план более лёгкий, фокус на технику и кардио.
- Пользователь поменял частоту с 3 на 5 → флаг `plan_rebuild_required`, на следующем входе CTA «План устарел, перегенерировать?».
- ML-модель вернула невалидный план (например, дубликаты или несоответствие `equipment_available`) → срабатывает валидатор, генерация повторяется до 3 раз, потом fallback на rule-based.
- Inference-сервис недоступен → fallback rule-based из встроенных шаблонов по уровню/цели/частоте.

## 11. Out of Scope

- Кастомизация сплита пользователем (push/pull/legs vs upper/lower) — в MVP модель выбирает сама
- Альтернативы упражнения inline («заменить»)
- Видео-демонстрация
- A/B-тестирование разных моделей внутри продакшена
- Цикл периодизации длиннее 4 недель (мезоциклы)
- Учёт возраста травм / противопоказаний — отдельная story в фазе 2 (требует медицинской валидации)
- Подсказки от ML по технике (CV-анализ видео из тренировки)
