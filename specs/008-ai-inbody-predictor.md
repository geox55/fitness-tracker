# Specification: AI прогноз изменений InBody (своя ML-модель)

**Epic:** E4 — AI-блок Маши
**User Story:** Пользователь должен видеть прогноз изменений своего InBody (вес, % жира, мышечная масса) на 1–4 недели вперёд, чтобы понимать, к чему он идёт при текущих тренировках и питании.
**Related specs:** [003-inbody-measurements.md](003-inbody-measurements.md), [005-workout-tracking.md](005-workout-tracking.md), [007-ai-nutrition-planner.md](007-ai-nutrition-planner.md), [009-plan-adaptation-and-chat.md](009-plan-adaptation-and-chat.md), [010-analytics-reports.md](010-analytics-reports.md), [012-ml-dataset.md](012-ml-dataset.md)

---

## 1. User Goal

Пользователь хочет понимать, эффективна ли его текущая стратегия. Не «как я выгляжу сейчас», а «куда я иду через 4 недели при текущем плане». На основе истории InBody, тренировок и питания собственная ML-модель прогнозирует:
- вес,
- процент жира,
- мышечную массу.

Прогноз показывается с доверительным интервалом и обновляется по мере накопления новых данных.

Это **главный AI-блок Маши** для диплома: своя ML-модель, обучаемая на объединённом датасете (см. [012](012-ml-dataset.md)).

---

## 2. Context

Постановка задачи: time-series регрессия. На вход — фичи пользователя по неделям (агрегаты тренировок, питания, истории InBody, профиль). На выход — прогнозируемые значения InBody через t недель.

Подход (фиксируется в продуктовой спеке только как «что должно работать»):
- Базовый baseline: линейная регрессия по тренду InBody.
- Целевая модель: gradient boosting / небольшая RNN / LSTM по последовательности недель.
- Обучается на:
  1. Исторических данных пользователя (если их достаточно — ≥4 InBody).
  2. Кросс-пользовательский трансфер на синтетических/анонимизированных данных из dataset (см. [012](012-ml-dataset.md)).

Архитектура и метрики ML-части — в магистерской работе Маши.

## 3. User Scenarios

### Scenario 1 — Прогноз для пользователя с историей

Пользователь сделал 4+ InBody, регулярно тренируется и ведёт питание.

**How to test independently**: Создать пользователя с 4 InBody-замерами и 8 тренировками за 4 недели, запросить прогноз на 4 недели, проверить что вернулись три значения с CI.

**Acceptance criteria**:

1. **Given** у пользователя ≥4 InBody-измерений и ≥8 тренировок за последние 8 недель.
   **When** он открывает раздел «Прогноз» (или это вызывается автоматически после каждого нового InBody).
   **Then** возвращается прогноз: weight_kg, body_fat_percent, muscle_mass_kg на горизонтах +1, +2, +4 недели; для каждого — точечная оценка + 80%-доверительный интервал.

2. **Given** прогноз готов.
   **When** пользователь его смотрит.
   **Then** видит график: историческая линия + прогнозная линия (пунктирная) + ленту CI; короткая текстовая интерпретация («при текущем темпе — ещё −1.2 кг за 4 недели, ±0.5 кг»).

### Scenario 2 — Прогноз для нового пользователя (cold start)

У пользователя только 1 InBody, тренировок мало.

**Acceptance criteria**:

1. **Given** у пользователя <4 InBody-измерений.
   **When** запрашивается прогноз.
   **Then** возвращается оценка по cross-user baseline (на основе цели + InBody-снимка + типичной траектории), с явным флагом `confidence: low` и текстом «Прогноз станет точнее после ещё 2 измерений».

2. **Given** у пользователя 0 InBody-измерений.
   **When** запрашивается прогноз.
   **Then** возвращается ошибка `404 not_enough_data` с CTA «Добавьте InBody, чтобы увидеть прогноз».

### Scenario 3 — Сравнение прогноз vs факт

Когда приходит новый InBody, пользователь видит, насколько прошлый прогноз совпал.

**Acceptance criteria**:

1. **Given** ранее был сделан прогноз на 28 дней.
   **When** пришёл новый InBody на ~T+28 (±3 дня).
   **Then** в карточке нового измерения видно: «Прогноз был X кг ± Y, факт Z кг», и MAE/направление ошибки.

2. **Given** у Маши есть набор прошлых прогноз/факт-пар по всем пользователям.
   **When** считаются метрики качества модели.
   **Then** агрегаты MAE, MAPE, calibration логируются для дашборда модели (для магистерской).

### Scenario 4 — Прогноз "what-if" (опц.)

Пользователь хочет понять: «а если я буду тренироваться 5 раз в неделю вместо 3?»

**How to test independently**: Запросить прогноз с override-параметрами, сравнить с базовым прогнозом.

**Acceptance criteria**:

1. **Given** активный план + история.
   **When** пользователь меняет в UI «частота тренировок» на 5 (без сохранения в профиль) и нажимает «Пересчитать прогноз».
   **Then** возвращается альтернативный прогноз; UI показывает обе линии (текущая стратегия / альтернативная).

2. **Given** override-параметр некорректен (frequency=10).
   **When** запрос отправлен.
   **Then** ошибка валидации `400`.

## 4. Functional Requirements

| #      | Requirement                                                                                                                                  |
|--------|----------------------------------------------------------------------------------------------------------------------------------------------|
| REQ-01 | Прогноз трёх метрик: weight_kg, body_fat_percent, muscle_mass_kg                                                                              |
| REQ-02 | Горизонты прогноза: +1, +2, +4 недели от текущей даты                                                                                          |
| REQ-03 | Возврат точечной оценки + 80% доверительного интервала                                                                                        |
| REQ-04 | Определение `confidence`: `high` \| `medium` \| `low` в зависимости от количества данных                                                       |
| REQ-05 | Короткая текстовая интерпретация (1-2 предложения), сгенерированная rule-based                                                                |
| REQ-06 | Использование фичей: история InBody, тренировки (тоннаж, частота, длительность), питание (target_calories, фактическое потребление если есть), профиль |
| REQ-07 | Cross-user baseline для cold-start пользователей (<4 InBody)                                                                                  |
| REQ-08 | Возврат ошибки 404 при <1 InBody                                                                                                              |
| REQ-09 | Логирование прогнозов для последующего сравнения с фактом и дообучения                                                                        |
| REQ-10 | What-if прогноз с override-параметрами (training_frequency, calories_offset)                                                                  |
| REQ-11 | Версионирование модели (`model_version` в каждом ответе)                                                                                      |
| REQ-12 | Fallback: если ML-инференс упал → линейный baseline по тренду InBody, с флагом `fallback: true`                                                |

## 6. Success Criteria

| #     | Criterion                                                          | How to measure                                                                |
|-------|--------------------------------------------------------------------|-------------------------------------------------------------------------------|
| SC-01 | MAE прогноза веса на горизонте 4 недели ≤1.5 кг (high-confidence)  | Replay на исторических данных тестовой выборки                                |
| SC-02 | MAE прогноза body_fat_percent ≤1.5 пп (high-confidence)            | Replay                                                                        |
| SC-03 | Доверительный интервал калиброван: фактическое значение попадает в 80%-CI ≥75% случаев | Coverage на тестовой выборке                                |
| SC-04 | Модель превосходит линейный baseline по MAE на ≥10%                | Сравнение моделей на одной тестовой выборке                                   |

## 8. Screens

### Screen: Прогноз

**Purpose**: Дать пользователю взгляд в будущее.

**Elements**:
- Заголовок «Прогноз на 4 недели»
- Три карточки (вес, % жира, мышечная масса) с числами +1н / +2н / +4н и стрелками тренда
- График с исторической линией + прогнозом + лентой CI
- Короткий текст-интерпретация
- Бейдж confidence
- (Опц.) Кнопка «What-if» → форма override

**States**:
- Empty: «Добавьте хотя бы одно InBody-измерение»
- Loading: спиннер
- Low confidence: жёлтый баннер «Прогноз станет точнее после ещё 2 измерений»

### Screen: Сравнение прогноз vs факт

(Встроено в карточку нового InBody-измерения, см. [003](003-inbody-measurements.md).)

**Elements**: «Прогноз был ... ± ... | Факт ...» + знак ✅/⚠️ попал ли факт в CI.

## 9. API Endpoints

### Получить прогноз

```
GET /api/v1/forecast/inbody?horizons=1,2,4
```

**Response 200**:
```json
{
  "generated_at": "2026-04-28T12:00:00Z",
  "model_version": "inbody-pred-0.4.2",
  "based_on_inbody_id": "uuid",
  "confidence": "high",
  "fallback": false,
  "metrics": {
    "weight_kg": [
      { "horizon_weeks": 1, "point": 77.6, "ci_low": 77.0, "ci_high": 78.2 },
      { "horizon_weeks": 2, "point": 77.0, "ci_low": 76.0, "ci_high": 78.0 },
      { "horizon_weeks": 4, "point": 76.2, "ci_low": 74.7, "ci_high": 77.7 }
    ],
    "body_fat_percent": [...],
    "muscle_mass_kg": [...]
  },
  "interpretation": "При текущем темпе вы потеряете ~1.2 кг за 4 недели, преимущественно за счёт жира."
}
```

**Errors**:
| Code | Situation                                  |
|------|--------------------------------------------|
| 404  | Нет ни одного InBody-измерения             |

### What-if

```
POST /api/v1/forecast/inbody/what-if
```
**Request**:
```json
{
  "horizons": [1, 2, 4],
  "override": {
    "training_frequency": 5,
    "calories_offset_kcal": -200
  }
}
```

**Response 200**: то же, что и GET, но с пометкой `what_if: true`.

### История прогнозов (для дашборда модели)

```
GET /api/v1/forecast/inbody/history?limit=20
```
**Response 200**: список прогнозов с проставленными ForecastEvaluation, если факт уже пришёл.

## 10. Edge Cases

- 1 InBody — прогноз есть, но `confidence: low` и предупреждение.
- Очень короткая история (2-3 InBody за неделю) — модель не должна экстраполировать резкие тренды; ограничивает скорость изменения по физиологическому максимуму (например, ≤1.5 кг/неделю).
- Большой пропуск (нет InBody >60 дней) — confidence снижается, прогноз пересчитывается на текущий момент только после нового измерения.
- ML-сервис недоступен → fallback линейный baseline.
- Изменение цели (weight_loss → muscle_gain) обнуляет confidence до `medium` минимум на 4 недели.

## 11. Out of Scope

- Прогноз других показателей (висцеральный жир, % воды) — фаза 2
- Прогноз на горизонт >4 недель
- Уведомления о расхождении прогноз/факт пользователю — отдельная story в [011](011-notifications.md)
- Causal inference (объяснение «почему»: какой фактор влияет) — фаза 2 / магистерская расширение
- Прогноз травм / здоровья — out of scope, требует медицинской валидации
