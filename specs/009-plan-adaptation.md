# Specification: Адаптация плана

**Epic:** E4 — AI-блок Маши
**User Story:** Пользователь должен получать рекомендации по корректировке плана тренировок при изменении веса/цели/частоты.
**Related specs:** [002-user-profile.md](002-user-profile.md), [003-inbody-measurements.md](003-inbody-measurements.md), [006-ai-workout-generator.md](006-ai-workout-generator.md), [008-ai-inbody-predictor.md](008-ai-inbody-predictor.md)

---

## 1. User Goal

Пользователь не должен сам решать, когда пора пересобрать план тренировок. Если он изменил цель, увеличил частоту, заметно потерял или набрал вес — приложение замечает это и либо автоматически пересобирает план, либо предлагает пересобрать.

Это сервис-watcher над сигналами:
- Новый InBody-замер → проверяет дельту с предыдущим.
- Изменение поля профиля (`goal`, `training_frequency`, `equipment_available`).
- Прошло ≥4 недели с генерации плана.

При срабатывании триггера — либо автоматически перегенерирует, либо ставит флаг `plan_rebuild_required` и показывает баннер на главном экране.

## 2. User Scenarios

### Scenario 1 — Автокоррекция при изменении веса

Пользователь добавил InBody-замер: вес изменился значительно (>2 кг или >3% от предыдущего за <30 дней).

**How to test independently**: Создать пользователя с активным планом, добавить новый InBody с дельтой 3 кг от последнего, проверить что в `plan_rebuild_events` появилась запись с `trigger='weight_change'`.

**Acceptance criteria**:

1. **Given** есть активный план тренировок и приходит новый InBody.
   **When** дельта `|new.weight_kg − prev.weight_kg| > 2 кг` ИЛИ изменение >3% за <30 дней.
   **Then** в `plan_rebuild_events` создаётся запись с `trigger='weight_change'`, `status='pending'`; на главном экране появляется баннер «План тренировок устарел, перегенерировать?»; ставится флаг `plan_rebuild_recommended`.

2. **Given** дельта незначительна (<2 кг и <3%).
   **When** приходит новый InBody.
   **Then** план не меняется, обновляется только прогноз ([008](008-ai-inbody-predictor.md)).

### Scenario 2 — Регенерация при смене цели или частоты

Пользователь сменил цель с «похудение» на «набор массы».

**Acceptance criteria**:

1. **Given** активный план тренировок.
   **When** пользователь меняет в профиле `goal` или `training_frequency`.
   **Then** план тренировок помечается `plan_rebuild_required: true`; пользователь видит баннер на главном экране «План устарел, перегенерировать?» с CTA.

2. **Given** пользователь подтверждает регенерацию.
   **When** он нажимает CTA.
   **Then** старый план архивируется, новый генерируется по обновлённым параметрам.

3. **Given** пользователь игнорирует баннер 7 дней.
   **When** проходит фоновая проверка.
   **Then** регенерация запускается автоматически, баннер обновляется на «План перегенерирован».

### Scenario 3 — Окончание 4-недельного цикла

**Acceptance criteria**:

1. **Given** план тренировок: `valid_until` ≤ today.
   **When** проходит фоновая проверка (раз в сутки).
   **Then** план архивируется; ML-модель ([006](006-ai-workout-generator.md)) генерирует новый с учётом данных прошедшего цикла (фактических подходов/повторений из тренировок).

2. **Given** в прошлом цикле пользователь стабильно превышал целевые повторения.
   **When** генерируется новый план.
   **Then** в новом плане увеличены целевые веса/повторения (прогрессия от факта).

## 3. Functional Requirements

| #      | Requirement                                                                                                                |
|--------|----------------------------------------------------------------------------------------------------------------------------|
| REQ-01 | Watcher на изменение InBody.weight_kg: триггер при дельте >2 кг или >3% за <30 дней                                         |
| REQ-02 | Watcher на изменение полей профиля: `goal`, `training_frequency`, `equipment_available` → ставит `plan_rebuild_required`     |
| REQ-03 | Watcher на окончание 4-недельного цикла: автоматически архивирует и регенерирует                                          |
| REQ-04 | Принудительная регенерация по флагу через 7 дней игнорирования                                                              |
| REQ-05 | Регенерация плана тренировок учитывает фактические данные предыдущего цикла (тоннаж, RPE)                                   |
| REQ-06 | Архивирование старого плана сохраняется в истории                                                                            |

## 5. Data Model

**Entity: PlanRebuildEvent** (audit watcher'а)

| Field        | Type      | Constraints                                              | Description                                                                |
|--------------|-----------|----------------------------------------------------------|----------------------------------------------------------------------------|
| id           | UUID      | PK                                                       |                                                                            |
| user_id      | UUID      | FK → User, required                                      |                                                                            |
| trigger      | Enum      | `weight_change` \| `goal_change` \| `frequency_change` \| `cycle_end` \| `manual` | |
| target_plan  | Enum      | `workout`                                                | В MVP только тренировки                                                    |
| status       | Enum      | `pending` \| `auto_applied` \| `user_confirmed` \| `dismissed` | Подтвердил ли пользователь                                            |
| triggered_at | Timestamp | Required                                                 |                                                                            |
| applied_at   | Timestamp | Optional                                                 |                                                                            |

## 7. API Endpoints

### Триггеры адаптации (внутренние)

Эти эндпоинты вызываются другими сервисами, не пользователем:
```
POST /internal/v1/adaptation/check-weight    { "user_id": "..." }
POST /internal/v1/adaptation/check-cycles     (cron, без user_id, проходит по всем)
```

### Подтверждение регенерации (пользователем)

```
POST /api/v1/plans/rebuild
{ "target": "workout" }
```

### События адаптации

```
GET /api/v1/adaptation/events?status=pending
```

## 8. Edge Cases

- Пользователь добавил два InBody подряд за один день → сравнивается с предыдущим из другого дня, не друг с другом.
- Изменение `goal` происходит несколько раз за день → debounce: rebuild_required ставится один раз, регенерация не запускается каждый раз.
- Пользователь удаляет аккаунт → события адаптации удаляются вместе с пользователем (GDPR).
